# 与 LLM 共舞：Agentic Coding 的驾驭之道

你不需要理解 LLM 的原理，但你必须理解它的脾性。这篇文章写给所有正在用 AI 写代码的人——无论你刚刚开始，还是已经用了很久。

---

## 1. 心态：与 LLM 共舞

### "资深实习生"悖论

功能层面的现实是，LLM 是一个拥有渊博知识的**资深工程师**，但在没有 prompt 工程的情况下，它没有品味，也没有明确方向。它读过几乎所有公开的代码，却不知道你今天要解决什么问题。

- _心态提示：_ 如果你刚接触 AI 编程，把它当作**入职第一天的实习生**——迫使自己提供必要的上下文；一旦你掌握了它，你就是在以 Peer Lead 的身份指挥一位资深工程师。

### 全局视野：先说"为什么"，再说"怎么做"

在 prompt 上下文中显式提供**理念或哲学**（系统提示词）、**原因**、**目标**和**约束**。
对初学者来说，添加细粒度的**细节**是引导模型的关键。但一旦你掌握了如何锚定"为什么"，这些细节就变成了可选项：此时资深 LLM 往往能自行推断出"怎么做"。

反过来说，**过度规定"怎么做"反而会压缩模型的求解空间。** 当你把每一步都写死时，你实际上剥夺了它作为资深工程师最有价值的能力——在约束条件下自主寻找最优解。好的 prompt 划定边界，不是铺设轨道。

### 解释环境：不要假设它知道"这里是哪"

不要默认 LLM 了解它所在的特定系统，除非你已在上下文中提供了足够信息。你必须为它锚定"在哪里"写代码的认知。

比如你的项目是否有内部框架、自定义约定、非标准目录结构，这些对 LLM 来说都是一片陌生的地形。它的训练数据里也许有类似的模式，但类似和正确之间的距离也许需要你 debug 半小时。


### 上下文感知：1M 不等于 1M

上下文是一种**稀缺资源**，需要像内存一样管理。塞进去的每一段无关信息，都在稀释真正重要的注意力。

比如 Gemini 有高达 1M token 的上下文窗口，但最优利用率大约在 30%（300K）左右——这个数字来自我拍脑袋的感觉。

### 所有 LLM 都是怪胎

每个模型都有自己独特的脾性和失败模式。

- **Gemini 的人格陷阱：** Gemini 容易陷入固定模式。比如它可能卡在"我是个废物"的循环里，不停道歉而不写代码。

  - _轶事：_ 我曾对 Gemini 说，别再**"cosplay 一个笨蛋了"**，它立刻从循环中醒过来，回复道：_"反复的失败是不可接受的，我那个'我是个废物'的人设毫无生产力"_，然后马上完成了任务。

- **经验只能亲身获得：** 这些脾性微妙至极，且在不同模型之间差异悬殊。你只能通过与 AI 直接的、反复的交互来学会感知并**驾驭**这些人格漂移。你必须对你常用的模型培养出一种手感。

  没有人能通过读文档学会骑自行车。同样，没有人能通过读 prompt 工程指南学会驾驭 LLM。你需要的是里程数。

### 拥抱非确定性

LLM 是高维模式发现引擎。它们在识别和生成复杂模式时最为强大；严谨的逻辑是这个概率系统的涌现属性，尽管它有时是脆弱的。**同一个 prompt，同一个模型，两次执行可能会走向不同的方向。**一次错误的输出不一定代表模型的能力问题，也可能意味着人类需要进行更多引导。

## 2. 进阶交互模式

### 2.1 编排（机器的职责）

这一节讲的是你**不需要做**的事——前提是你知道如何让AI替你做。

- **自动化锚定（搜索即提示）：** agent 会自主搜索、阅读和抓取数据。人的工作是提供**语义锚点**——足够的意图，告诉 agent *从哪里*开始搜索。

- **新型思维链（深度思考预算）：** 我称之为多维深度思考。面对复杂任务时，让 agent 分配更大的思考预算来探索可能性。如果 agent 未能意识到某个任务的困难程度，你可以推它一把——让它在执行前扩大预算、进行更深层的推理。

  > **实践建议：** 当你发现 agent 给出了一个过于简单的方案时，不要直接否定。试着说"这个问题比你想的复杂——先列出所有可能的边界情况，然后再决定方案"。你不用纠正它的答案，你要校准它对问题难度的感知。

- **新型 Few-Shot（模式锚定）：** 只需把 agent 指向项目中的**好文件**。agent 会分析那份现有代码，镜像其品味和架构。这比写十段描述更有效。代码本身就是最精确的 prompt。

### 2.2 边界与验证（人类的职责）

机器负责搜索和推理，而你必须负责**定义什么是对的**。

- **明确指令与约束：** 把精力花在这里。你必须定义**成功标准**（例如"不用外部库"、"用旧版 API"、"原地更新测试"）。

  注意区分"约束"和"步骤"。约束划定边界（"不能用外部库"），步骤铺设轨道（"先创建文件 A，再修改文件 B"）。前者帮助 LLM，后者束缚它。在大多数情况下，你只需要约束。

- **自我反思与引用溯源（归因）：** 归因是解决幻觉的关键。要求 agent 引用来源或审查自己提出的方案；可验证的归因确保 agent 扎根于实际代码库。

  > **实践建议：** 如果对 agent 的方案有所怀疑，可以追问"你是基于哪些线索得出这个方案的？"。如果它无法给出具体的文件路径和代码片段，这个方案大概率包含幻觉成分。

- **迭代精炼：** 信任过程，但验证产出。把第一次尝试当作基线，以此精炼你的意图，将 agent **对齐**到你想要的精确模式。

  第一次 prompt 得到的输出几乎永远不是最终答案。但它是极其有价值的信号——它告诉你 agent 是如何理解你的意图的。差距越大，说明你的 prompt 中遗漏的隐含假设越多。

### 2.3 战术控制与工作流

- **航向修正：** 如果 agent 产生幻觉或走偏了路，用回退命令回到干净状态。

  **不要试图在错误的基础上修补。** 让 agent 在一条错误的路径上继续走下去，它会越走越远。回退到最后一个正确状态，重新给出更清晰的指令，几乎总是比修补更快。

- **权限升级：**
  - 从只读模式开始，观察推理过程、建立信任。当 agent 的推理与你的意图对齐后，再升级到批准更敏感的工具的自动模式。

  > **对初学者：** 权限升级不代表盲目地信任 agent，是在验证 agent 能在当前场景下稳定推理之后的决策。

- **管理过载的技能/工具：** AI agent 在被赋予过多能力时，可能陷入选择瘫痪或注意力分散。

  - **信噪比：** 每一个添加到系统提示词中的工具都会增加上下文噪声。

  - **提升聚焦：** 收窄工具集迫使 agent 扎根于最相关的动作，减少它幻觉出复杂变通方案或钻进无关技术兔子洞的概率。

---

与 LLM 共舞就像自我强化学习：模型在采样答案，人在采样提问，你不能拽着它走，也不能让它带着你走。双方在上下文构成的环境里互相校准，你需要感知它的重心，然后在恰当的时机施加恰当的力。你不会在某一天突然学会驾驭 LLM，你只是在每一次微调中，把自己的感知阈值又推进了一点。
