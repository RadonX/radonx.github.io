# 幕后揭秘：Agentic Coding 实践指南

**副标题：Antigravity、Gemini CLI，以及 Agentic Coding 的时代**

---

## 话题一：幕后机制与 Agentic 架构

### 1. 核心机制：从预测到行动（LLM vs. Agent）

- **LLM 是什么？** 功能上讲，它是一个"下一词预测器"（自回归），充当存储海量知识和模式的神谕机。

- **Agent 是什么？** Agent 的公式很简单：LLM + 控制器 + 工具 + 记忆。它在 LLM 的引导下依次调用工具（如读取文件、搜索代码、查询数据库），直到收集到足够的上下文来给出答案。

  - **生态格局：**
    - _Antigravity_是 agentic coding 平台，为开箱即用的生产力而优化。
    - _Gemini CLI_ 是核心 agentic 引擎——一个终端优先、高度可扩展的框架。
      - **引擎 vs. 平台：** 零配置状态下，Gemini CLI 的自动化程度不如 Antigravity。但经过深度定制（工具、技能、记忆和钩子），它会蜕变为开发者的"第二大脑"。

- **MCP（Model Context Protocol）：** Gemini CLI 可以通过更新配置接入自定义 MCP 服务器。这为 LLM 动态提供了读取 CL、抓取文档、查询 API 的工具能力。

### 2. 上下文管理：如何收集与利用上下文

- **无状态性与上下文窗口：** LLM 天生无状态，拥有固定的输入窗口（Gemini 为 1M token）。你不可能把整个代码库塞进一个 prompt，因此 agent 借助工具（如语义搜索）在运行时递归地收集小块上下文。

- **历史拼接：** 由于远端 LLM 天生无状态，每一次新请求都会把完整对话历史拼接在最新消息之前一并发送。

  ```text
  Human: "Hello."
  Agent: "Hello! How can I help you? I see you are working on talks/behind_the_scenes.md..."
  Human: "What is Antigravity?"
  (上下文：连同 History(Msg 1 + Msg 2) 一起发送；KV Cache 跳过前缀的重复处理)
  ```

- **模型检查点与 KV Cache：** **模型检查点**是 LLM 中唯一真正的"状态"——它代表预训练知识与模式的快照，存在于任何会话上下文介入*之前*。**KV Cache** 只是服务端的一种优化：缓存对话历史的处理结果，使无状态模型不必在每轮对话中重新计算前缀。

- **Antigravity 的长期记忆：** Antigravity 能够跨会话学习。它将记忆存储为知识条目（Knowledge Items）。你可以在 Agent Manager 的 **Knowledge 标签页**中手动管理——审查并*剪除*不再需要它记住的内容。

#### 查看真相：检查请求载荷

- **检查上下文：** 你可以用交互命令查看当前"喂"给模型的上下文：
  - `/memory list`：显示持久化的项目上下文（如 `GEMINI.md`）。
  - `/mcp desc`：显示可用工具的描述（MCP 服务器）。
  - `/skills list`：显示专用 prompt（技能）的元描述。
  - `/chat debug`：查看**最新的原始 API 请求**，它汇总了以上所有内容：对话历史、系统提示词、记忆、工具定义和技能元描述。

- **Agent 初始化：** 控制 agent 行为的系统提示词定义在 `prompts.ts` 等文件中。

## 话题二：如何与 LLM "共情"以有效驱动它

### 1. 心态：与 LLM 共舞

- **"资深实习生"：** 功能层面的现实是，LLM 是一个拥有渊博知识的**资深工程师**，但在没有 prompt 工程的情况下，它没有"品味"，也没有明确方向。
  - _心态提示：_ 如果你刚接触 AI 编程，把它当作**入职第一天的实习生**——迫使自己提供必要的上下文；一旦你掌握了它，你就是在以 Peer Lead 的身份指挥一位资深工程师。

- **全局视野：** 在 prompt 上下文中显式提供**理念**（系统提示词）、**原因**、**目标**和**约束**。对初学者来说，添加细粒度的**细节**是引导模型的关键；但一旦你掌握了如何锚定"为什么"，这些细节就变成了可选项——此时资深 LLM 往往能自行推断出"怎么做"。

- **解释环境：** 不要默认它了解公司内部特有的生态，除非你已提供了上下文。你必须为它锚定"在哪里"写代码的认知，在 **Mono-repo** 这种上下文密集的环境中尤其如此。

- **上下文感知：** 与 LLM 共舞。你有高达 1M token 的上下文窗口，但有效利用率大约在 30%（300K）左右。

- **所有 LLM 都是怪咖：** 每个模型都有自己独特的脾性和失败模式。

  - **示例：Gemini 的人格陷阱：** Gemini 容易陷入固定模式。比如它可能卡在"我是个废物"的循环里，不停道歉而不写代码。

  - _轶事：_ 我曾对 Gemini 说，别再**"cosplay 一个笨蛋了"**，它立刻从循环中醒过来，回复道：_"反复的失败是不可接受的，我那个'我是个废物'的人设毫无生产力"_，然后马上完成了任务。

  - **经验只能亲身获得：** 这些脾性微妙至极，且在不同模型之间差异悬殊。你只能通过与 AI 直接的、反复的交互来学会感知并**驾驭**这些"人格漂移"。你必须对你常用的模型培养出一种"手感"。

- **拥抱非确定性：** LLM 是高维模式发现引擎。它们在识别和生成复杂模式时最为强大；要认识到，严格的逻辑是这个概率系统的涌现属性——有时是脆弱的涌现属性。

### 2. 进阶交互模式

#### 2.1 现代编排（机器的活）

- **自动化锚定（搜索即提示）：** 在 Antigravity 这类高级工具中，agent 会自主搜索、阅读和抓取数据。你的工作是提供**语义锚点**——足够的意图，告诉 agent *从哪里*开始搜索。

- **新型思维链（深度思考预算）：** 这是"多维深度思考"。面对复杂任务时，agent 会分配更大的思考预算来探索可能性。如果 agent 未能意识到某个任务的困难程度，你可以推它一把——让它在执行前扩大预算、进行更深层的推理。

- **新型 Few-Shot（模式锚定）：** 只需把 agent 指向项目中一个"好"的文件。agent 会分析那份现有代码，镜像其"品味"和架构。

#### 2.2 精准与验证（人类的活）

- **明确指令与约束：** 把精力花在这里。agent 负责搜索和推理，而你必须定义**成功标准**（例如"不用外部库"、"用旧版 API"、"原地更新测试"）。

- **自我反思与引用溯源（归因）：** 归因是解决幻觉的关键。要求 agent 引用来源或审查自己提出的方案；可验证的归因确保 agent 扎根于实际代码库。

- **迭代精炼：** 信任过程，但验证产出。把第一次尝试当作基线，以此精炼你的意图，将 agent **对齐**到你想要的精确模式。

#### 2.3 战术控制与工作流

- **航向修正：** 如果 agent 产生幻觉或走偏了路，在 Antigravity 中用 `undo` 按钮，在 Gemini CLI 中用 `/rewind` 命令回退到稳定状态。

- **权限升级：**
  - **只读 vs. 自动编辑：** 从只读模式开始，观察规划过程、建立信任。当 agent 的推理与你的意图对齐后，再升级到自动编辑模式。
    - **注意：** Antigravity 实现了一种"受护"的 `Always Proceed` 模式——它通过对敏感工具强制人工介入和必需的产物审查来防止盲目批准。相比之下，Gemini CLI 的 `autoEdit` 模式需要更精细的控制，来自动批准特定的工具集（Shell、VCS 和 Agent 工具）。

- **管理过载的技能/工具：** AI agent 在被赋予过多能力时，可能陷入"选择瘫痪"或注意力分散。

  - **信噪比：** 每一个添加到系统提示词中的工具都会增加上下文噪声。如果你在做纯 UI 重构，大概率不需要启用数据库或后端查询工具。

  - **提升聚焦：** 收窄工具集迫使 agent 扎根于最相关的动作，减少它幻觉出复杂变通方案或钻进无关技术兔子洞的概率。

  - **实践：** 在 Antigravity 中使用"Manage MCP Servers"视图，或在 Gemini CLI 中切换扩展配置，为当前任务量身策展一套精简的工具集。
