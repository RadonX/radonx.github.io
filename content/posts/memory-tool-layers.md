## 遗忘

我的 agent 把我忘了。

不是隐喻。周一我们花了两个小时讨论一个部署方案，周二它问我是不是第一次用 OpenClaw。Session reset 之后，一切归零。它甚至没有那种忘记了什么的不安。它不是忘了，它是从来不知道自己应该记得。

于是我开始在社区里翻，想给它装上某种长期记忆。Nocturne、lossless-claw、QMD、ClawDB、Sirchmunk、MemOS——名字一个比一个陌生，README 一个比一个自信。每个项目都在说自己解决了记忆问题，但当我试图把它们放在一起比较的时候，发现比不了。有人说 Nocturne 比 ClawDB 好，有人说反过来，两边的论据都无懈可击，结论却完全相反。

我在这种矛盾里转了好几天，直到某个晚上突然想明白：不是答案矛盾。是我一直在拿螺丝刀和锤子比较谁更锋利。

## 一个词，三件事

问题出在**记忆**这两个字上。

它太大了。就像你说一个人性格好，到底是说他温柔、说他守信、还是说他在你搬家的时候会来帮忙？这三件事之间几乎没有交集，但它们都可以被叫做性格好。

在 agent 系统里也是一样。我们说记忆的时候，其实在同时谈论三件毫无关系的事情。

第一件是**身份**。agent 应该遵守什么原则、回避什么话题、在什么情况下用什么语气。这些东西不会随着对话变化，它们更像是性格，而不是经历。

第二件是**经历**。我们上周聊了什么，做了什么决定，在哪个问题上吵了半天最后选了方案 B。这些东西会随时间增长，会过期，会被后来的决定推翻。

第三件是**知识**。某个配置参数的默认值是什么，那篇笔记放在哪个文件夹里，那段代码是谁在什么时候改的。这些东西需要的不是回忆，是检索——带路径、带行号、带出处的那种。

这三件事需要完全不同的存储方式、调用时机、失败策略。但因为它们共享了同一个名字，它们被塞进了同一个抽屉。结果就是 RAG 开始冒充人格，摘要开始冒充事实，相似度匹配开始冒充原则。每一步看起来都说得过去，但整个系统慢慢变成了一团无法调试的雾。

## 一刀

调研中最花时间的部分，不是比较工具，而是理解对话轨迹这一层里面还藏着什么。

我原本以为对话记忆就是一件事：存下来，找回来。后来才发现这里面有两个机制，它们的成本结构完全相反，像同一个房间里住着两个互相讨厌的人。

一个是**上下文引擎**。它在每一轮对话里都要出场，负责决定 prompt 怎么拼、旧消息留多少、摘要做到什么粒度。它每动一下，prefix 缓存就碎一次，延迟和成本跟着涨。它是一个安静不下来的机制。

另一个是**记忆后端**。它只在被点名的时候才出场——agent 主动调了 `memory_search`，它才开始干活。不调就不花一分钱。它是一个可以沉默的机制。

把这两个混在一起的后果，我在自己的账单上见过。配了 lossless-claw 之后，token 用量翻了将近一倍，但 agent 并没有变得更聪明，它只是每轮都在重新翻看所有的旧对话，像一个考试前把整本书从头读一遍的学生。问题不是出在 lossless-claw 本身——它是一个很好的上下文引擎——而是出在我把它当成了记忆后端来用。

想通这一刀之后，分层就不再是一个需要设计的东西了。它是问题本身的形状。

## 四层

**L0** 管身份。原则、偏好、禁忌。它不检索任何东西，它只裁决：这件事该不该做，那个话题该不该碰。在所有行动之前，它先画出边界。

**L3** 管经历。拆成两半：L3a 是上下文引擎，每轮都在场，负责 prompt 的拼装和压缩；L3b 是记忆后端，按需出场，负责回答 agent 的主动查询。

**L2** 管本地知识。我的笔记里有没有这个，那篇文档里第几行写了那句话。它输出的不是答案，是证据——path、snippet、position。

**L1** 管外网。成本最高，不确定性最大，是最后的升级路径。

| 层 | 管什么 | 什么时候动 | 为什么这个顺序 |
| :--- | :--- | :--- | :--- |
| L0 | 身份 | 永远第一个 | 不知道自己是谁就不该开始找东西 |
| L3 | 经历 | 第二个 | 最便宜、最相关 |
| L2 | 本地知识 | 第三个 | 内部证据优先于外部 |
| L1 | 外网 | 最后 | 贵，而且不一定可信 |

编号和执行顺序是反的。L1 编号最小但最后执行，L3 编号最大但最先执行。编号是按信息的通用性排的——外网最通用，对话轨迹最私有。执行顺序是按成本排的——从最便宜的开始。这不矛盾，只是两套不同的坐标系。

## 各就各位

理解了分层之后，那些工具各自回到了自己该在的位置，像散场后观众终于找到了自己的座位。

**Nocturne** 处理身份、原则、偏好，带快照回滚和审计。它的意义不是检索更强，而是让人格不再寄生在向量相似度上。它只能是 **L0**。

**lossless-claw** 做持久化消息、DAG 摘要、上下文拼装。它让旧对话不被简单截断，但代价是每轮都参与 prompt 构建，prefix 缓存因此变脆。它是 **L3a**，上下文引擎，不是通用记忆后端。

**memory-core、QMD、ClawDB** 长得很不一样——一个是 Markdown + SQLite，一个是本地 sidecar 带 BM25 和 rerank，一个是服务化后端带 WAL 和 Parquet。但它们回答的是同一个问题：agent 调了 `memory_search`，谁来接？它们都是 **L3b**。在这个位置上可以互相比较，离开这个位置比较就没有意义。

**Sirchmunk** 把检索变成了一串 agent 行为——query rewriting、evidence sampling、rerank。但它每次都要调 LLM，哪怕是 FAST 模式。如果输出不能稳定带上引用，它就不能进入事实链。它是 **L2**，本地知识层的增益。

**MemOS Cloud Plugin** 在每轮开始前自动翻旧账、注入 recall block，结束后写回。对陪伴型 AI 来说合情合理。但如果你想要一个安静的、可预测的系统，这种每轮自动注入的策略就不该是默认行为。

归个位：

- **L0 身份** — Nocturne
- **L3a 上下文引擎** — lossless-claw
- **L3b 记忆后端** — memory-core / QMD / ClawDB
- **L2 本地知识** — Sirchmunk
- **L1 外网** — tavily 等

## 克制

调研做到最后，我没有选出一个最强的工具。我选出了一种审美。

好的记忆系统是安静的。它不会每轮都自动拉历史、塞 recall block、改写上下文。大多数时候 agent 不需要回忆任何事情，它只需要把眼前的活干完。

好的记忆系统是可解释的。每一次 recall 都应该能交代清楚——为什么现在查，查到了什么，证据在哪。交代不清楚的 recall 就不该发生。

好的记忆系统是 cache-friendly 的。每轮都重写 prompt 结构的系统天然更贵、更慢、更难调试。这不是优化问题，是架构问题。

好的记忆系统是可审计的。尤其是 L0。约束是谁写进去的，什么时候改过，能不能回滚。如果人格层不可审计，你就永远不知道 agent 为什么突然换了一种口气跟你说话。

最后落地的默认策略反而是最简单的那个：L0 画边界，L3b 被叫到的时候才出场，L2 需要文档证据的时候才出场，L1 实在找不到了再去外网。大部分时候，L0 加按需的 L3b 就够了。其余的层，需要的时候再加，不需要就不要碰。

## 后来

写完这篇回过头想，记忆可能只是一个缩影。

在 agent 生态里，很多混乱看起来是技术选型的问题——该用哪个数据库、该选哪个框架、该不该上 RAG。但拆到底下去看，经常是同一个词在不同人嘴里指向了完全不同的东西，而大家在这个模糊的基础上争论、比较、做决策。

我现在觉得，在动手之前，最值得花时间的事情是把问题拆干净。搞清楚你到底在问几个问题，再去看每个工具回答的是哪一个。顺序反过来的话，你越研究越糊涂，因为你在一个不存在的维度上积累信息。

层位摆正，工具自然互补。层位混乱，再强的工具也只会互相污染。这件事大概不只适用于记忆。
