# 记忆工具分层：不是"谁更强"，而是"谁该在第几层"

## 混乱的根源

"记忆"这个词在 agent 系统里被严重滥用了。

当有人说"我的 agent 需要更好的记忆"，他可能在说三件完全不同的事：

- **我需要它记住自己是谁。** 它的性格、原则、我的偏好和禁忌——这些东西不该每次醒来都要重新学。
- **我需要它记住我们聊过什么。** 上周那个 bug 的结论，三天前做的决定，昨天讨论到一半的方案。
- **我需要它找到我写过的东西。** 某个配置写在哪篇笔记里，某个结论的原文出处，某份文档的具体段落。

这三类需求的性质完全不同。第一类是身份，第二类是轨迹，第三类是证据。它们需要不同的存储方式、不同的检索策略、不同的可靠性保证。

但市面上几乎所有关于"agent 记忆"的讨论，都把它们搅成一锅粥。用 RAG 来维持人格，用摘要来充当事实，用向量相似度来召回长期约束——每一个都是在用错误的工具解决错误的问题。

混乱不在于工具不够好，而在于问题没拆开。

## 拆开之后，分层自然浮现

一旦把"记忆"拆成身份、轨迹、证据三类，一个分层结构就会自然长出来。不是人为强加的，而是每类问题各自归位的结果。

**L0：身份层（Identity / Sovereign Memory）**

这一层处理的不是"信息检索"，而是"我是谁"。

它包括：agent 的性格设定、行为原则、用户的长期偏好、特定场景下的触发条件。它的输出不是"找到了一段相关文本"，而是"在这个情境下，我应该遵守什么约束"。

L0 不是检索器，它是路由器和裁判。它决定的不是"回忆什么"，而是"什么不能做"和"什么必须做"。

对 L0 最重要的要求不是智能，而是可审计——谁写进去的约束？什么时候改过？能不能回滚？如果你的人格层连修改历史都没有，那它就不是"主权记忆"，只是一堆随时可能被覆盖的配置。

**L3：对话轨迹层（Conversation / Runtime Trace）**

这一层处理的是"我们之前说过什么"。

但这里有一个关键的内部分界，如果不拆清楚，后果很严重：

- **L3a：上下文引擎（Context Engine）**——负责把历史对话压缩、组装成当前这轮 prompt 的上下文。它每轮都在工作，直接影响 prompt 结构。
- **L3b：记忆后端（Memory Backend）**——负责在 agent 主动调用 `memory_search` / `memory_get` 时，从索引里找到相关的记忆片段。它只在被叫到时才工作。

为什么必须拆？因为 L3a 和 L3b 的代价结构完全不同。

L3a 每轮都参与上下文组装，意味着 prompt 结构是动态的。动态 prompt 意味着 KV cache / prompt cache 的前缀无法稳定复用——每轮 prefill 成本上升，延迟变高，调试变难。这是一个持续的、隐性的成本。

L3b 是工具调用式的。agent 判断"我需要查一下之前的记录"，调用一次，拿到结果，用完就走。prompt 的核心结构不变，cache 命中率不受影响。

如果你把 L3a 和 L3b 混在一起，就会得到一个"既吵又贵又不可调试"的系统——它每轮都在偷偷改写你的上下文，你甚至不知道这轮的 prompt 和上一轮长什么不一样。

**L2：本地知识层（Local Knowledge Evidence）**

这一层处理的是"我写过的东西里有没有相关的"。

它面向的是本地文档库、笔记库、知识库。核心要求不是"会总结"，而是"能给引用"——返回的应该是 `path + snippet + position`，是证据，不是答案。如果一个检索工具只告诉你结论但不告诉你出处，它就不能进入事实链。

**L1：外网证据层（Web Evidence）**

这一层是最后的升级路径。成本最高，可靠性最低，不确定性最大。它应该是"内部找不到时才去外面找"，而不是默认入口。

## 把工具放回它们该在的位置

分层不是抽象游戏。它的价值在于：一旦你有了层位，现有工具该放哪就一目了然。

**Nocturne → L0**

Nocturne 解决的是身份、原则、偏好的管理，带 CRUD、快照回滚和审计功能。它不是用来"搜东西"的，而是用来"守规矩"的。它的价值不在于更强的检索，而在于：不再让人格和原则寄生在向量相似度上。

**lossless-claw → L3a**

lossless-claw 的核心能力是持久化消息、DAG 摘要、上下文组装。它让旧上下文不被粗暴截断，但代价是每轮都参与 prompt 拼装，prefix 稳定性下降，cache 复用变差。它是上下文引擎，不是通用记忆后端。

**memory-core / QMD / ClawDB → L3b**

这三者表面风格差异很大——memory-core 是内置默认（Markdown + SQLite），QMD 是本地 sidecar（BM25 + 向量 + rerank），ClawDB 是服务化后端（WAL / Parquet / DataFrame）。但从 OpenClaw 的抽象层看，它们都在实现同一对工具接口：`memory_search` 和 `memory_get`。都是 L3b。

**Sirchmunk → L2**

Sirchmunk 代表的是"把检索当成 agent 行为序列"——query rewriting、evidence sampling、summarization / rerank。它不是纯粹的索引器，但它需要 LLM API（哪怕是本地的），而且输出如果不能稳定变成"引用证据"，就不能进入事实链。它更适合做本地知识证据检索，而不是对话轨迹召回。

**MemOS Cloud Plugin → 不适合做默认层**

MemOS 的模式是"每次运行前自动搜旧记忆注入 prompt，结束后自动写回"。对某些陪伴型产品来说这合理，但如果你追求的是可预测、不吵、cache-friendly，那"每轮自动翻旧账"就是你最不想要的默认行为。

## 编号逻辑 ≠ 执行顺序

这套编号容易造成一个误解：L0 先执行，然后 L1、L2、L3？不是。

编号是按信息的**来源和性质**分类的。L0 是元信息，L1 是最通用的公开信息，L2 是私有策展信息，L3 是最具体的实例轨迹。

执行顺序是另一回事，它按**成本和相关性**决策：

| 编号 | 类别 | 执行顺序 | 为什么 |
| :--- | :--- | :--- | :--- |
| **L0** | Meta / 控制面 | **第 0 步** | 约束先于一切行动 |
| **L3** | 实例 / 轨迹 | **第 1 步** | 成本最低，时效性最高 |
| **L2** | 私有 / 策展 | **第 2 步** | 内部可信证据优先 |
| **L1** | 公开 / 外网 | **第 3 步** | 成本最高，最后升级 |

L0 排第一，是因为它与 agent 自身最相关——在做任何事之前，先知道自己的边界。然后是 L3（与当前任务最相关、成本最低），再 L2（需要文档证据时才查），最后 L1（实在找不到才去外网）。

这个顺序的本质是一条 escalation path：从最便宜、最可控的地方开始，逐步向更贵、更不确定的方向升级。

## 什么才算优雅

工具选完、层位摆好，还有一个问题：默认行为应该是什么？

四条标准：

**默认不吵。** 不应该每轮都自动拉历史、塞 recall block、改写上下文。安静是一种美德。

**默认可解释。** 每次 recall 都应该能回答三个问题：为什么现在查？查到了什么？证据在哪？如果回答不了，这次 recall 就不应该发生。

**默认 cache-friendly。** 如果你的系统每轮都重写 prompt 结构，它就天然更贵、更慢、更难调试。保持 prompt 结构的稳定性，是对成本和延迟最基本的尊重。

**默认可审计。** 尤其是 L0。约束是谁写进去的？什么时候改过？能不能回滚？如果不能回答这些问题，你的"主权记忆"就名不副实。

按这四条标准，优雅的默认策略其实很简单：

> L0 先决定约束。L3b 只在需要时被调用。L2 只在需要文档证据时被调用。L1 最后补外网。其余时间，系统保持安静。

## 每种工具背后是一种不同的信念

看清每个工具的"信念"，误用就会自动消失：

- **Nocturne 相信**：记忆首先是主权与身份。
- **lossless-claw 相信**：记忆首先是上下文压缩问题。
- **QMD / ClawDB 相信**：记忆首先是检索后端与工程可靠性问题。
- **Sirchmunk 相信**：检索是一种可自适应的 agent 行为，不是一次数据库查询。

它们都对。但它们回答的是不同的问题。把解决身份问题的工具拿去做检索，或者把解决检索问题的工具拿去维护人格，就是在错误的层位上使用正确的工具。

## 更好的问题

大多数记忆工具比较文章都会问：哪个效果更好？哪个更聪明？哪个更像长期记忆？

这些问题问错了。更好的问题是：

1. 它解决的是身份、轨迹、证据、还是外网搜索中的哪一类？
2. 它默认是自动注入还是按需调用？
3. 它输出的是答案还是证据？
4. 它的代价落在 cache、可解释性、审计性、部署复杂度中的哪一项？

一旦这样问，分层就会自然浮现，工具就会各归其位。

这不只适用于 OpenClaw。任何 agent 系统，只要遇到"大家把记忆说成一个词，但实际在讨论三四件不同的事"的现象，都可以用同一套步骤：先拆身份 / 轨迹 / 证据 / 外网，再看每个工具接在哪个接口上，再决定默认策略是否自动注入。

## 选型的最小决策树

- 你最在意**人格约束与审计** → 先做 L0（Nocturne）
- 你最在意**稳定、缓存、可预测** → L3b（memory-core / QMD / ClawDB）优先
- 你最在意**"丢文件就能搜"且接受 LLM 调用** → L2（Sirchmunk）做增益
- 你最在意**超长叙事连续性** → 再考虑 L3a（lossless-claw）作为特例

一句话：

> 记忆系统的关键，不是谁最强，而是谁该在第几层。层位摆正，工具互补；层位混乱，再强的工具也互相污染。

### L3b 后端选型矩阵

| 维度 | memory-core | QMD | ClawDB |
|---|---|---|---|
| Cache-Friendly | ✅ | ✅ | ✅ |
| 可解释性 | ✅ (SQLite) | ✅ (SQLite + explain) | ✅ (WAL 可回放) |
| 社区成熟度 | ✅ (内置) | 🟢 (活跃) | 🔴 (新) |
| 部署复杂度 | ✅ (零配置) | 🟠 (CLI/Bun/SQLite) | 🟠 (服务/队列) |

| 推荐默认 | ✅ | 🟢 (值得尝试) | 🟠 (PoC 优先) |

### 常见陷阱

**1. Cache 命中率悄悄崩了。** 元凶是 lossless-claw、MemOS 这类"每轮自动注入"的机制。prompt 结构变成动态拼装，prefix 不再稳定，每轮 prefill 成本上升。对策：默认用工具式 recall（L3b），保持 prompt 结构稳定。

**2. L0 和 L3 混用，人格开始飘。** 把 Nocturne 当资料库用，或者用 RAG 来检索行为原则。结果是 agent 行为不可预测，约束时有时无。对策：L0 只放"宪法"，不放"资料"；RAG 只检索"资料"，不碰"原则"。

**3. 双真相一致性地狱。** Markdown 文件和索引都被当成主存储，改了 MD 索引没跟上，或者反之。对策：明确"MD 为真相，索引为派生"，建立单向同步。

**4. 只读不写，agent 永远不长记性。** 只做 retrieval，忘了写回。agent 反复犯同一个错。对策：建立"失败 → 写 learnings → 定期 review → 提升到 L0 或 KB"的闭环。
