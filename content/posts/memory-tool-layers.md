## 一锅粥的起点

如果你在 OpenClaw 社区待过一段时间，一定见过这样的争论：

> "Nocturne 和 ClawDB 哪个好？"
> "lossless-claw 是不是比 QMD 更强？"
> "Sirchmunk 能不能替代 memory-core？"

我也曾经这样问。问完之后越来越糊涂，因为每个工具的作者都能给出令人信服的理由，但这些理由放在一起却互相矛盾。后来我意识到，不是答案矛盾，而是问题本身就问错了。

这些工具根本不在同一个维度上。把它们拉到一起比较，就像问"建筑师和水泥工谁更强"——答案取决于你在问什么问题。

---

## "记忆"这个词骗了我们

混乱的根源，是"记忆"这个词太大了。

在 agent 系统里，当我们说"记忆"的时候，其实至少在说三件完全不同的事情：

**第一件：我是谁。** 身份、原则、偏好、禁忌。你告诉 agent "永远不要用 var 声明变量"，这是记忆吗？是，但它和"上周那个 bug 的结论"完全不是一回事。

**第二件：我们聊过什么。** 对话轨迹、决策历史、上下文延续。上周你让 agent 帮你改了一个配置，这周想接着来——这也叫记忆，但它需要的是可回溯的对话痕迹，不是人格约束。

**第三件：我写过什么。** 文档、笔记、知识库里的某个具体细节。你知道答案在某篇笔记里，但想不起来在哪——这还是记忆，但它需要的是证据检索，带路径、带引用、带位置。

如果不先把这三件事拆开，任何记忆系统最终都会走向同一个结局：用 RAG 充当人格，用摘要充当事实，用相似度召回充当长期约束。每一步看起来都合理，但系统整体变成了一锅粥。

---

## 把层位摆正

拆清楚问题之后，分层就自然浮现了。不是我们"设计"了一个分层模型，而是问题本身就长成这个形状：

**L0：主权记忆层。** 处理身份、原则、偏好、禁忌。它不是检索器，而是裁判——在一切行动之前，先决定什么该做、什么不该做。

**L3：对话轨迹层。** 处理"我们之前聊过什么"。但这一层必须继续拆分，这是我们在调研中识别出的最关键的一刀：

- **L3a：上下文引擎。** 负责当轮 prompt 的组装——上下文压缩、摘要策略、对话拼接。
- **L3b：记忆后端。** 负责 `memory_search` / `memory_get` 的实际查询——当 agent 主动问"之前有没有聊过这个"时，去哪里找。

为什么必须拆？因为 L3a 每轮都参与 prompt 构建，它动一下，整个 prefix 缓存就失效了。而 L3b 是按需调用的工具，不调就不花成本。把这两者混在一起，你会得到一个既吵、又贵、又不可调试的系统。

**L2：本地知识证据层。** 处理"我的笔记/文档里有没有这个"。核心要求不是"会总结"，而是"能给引用"——path + snippet + position。

**L1：外网证据层。** 最后的升级路径。成本最高，不确定性最大，应该是最后才碰的一层。

---

## 编号和执行顺序是两回事

这里有一个容易绊住人的地方：为什么编号是 L0、L1、L2、L3，但执行顺序是 L0 → L3 → L2 → L1？

因为编号是按信息的**来源和通用性**排列的——L1 最通用（全网），L3 最具体（当前实例）。而执行顺序是按**成本和相关性**排列的——从最便宜、最相关的开始，逐步升级。

| 编号 | 类别 | 执行顺序 | 为什么 |
| :--- | :--- | :--- | :--- |
| L0 | 元层 / 控制面 | 第 0 步 | 约束先于行动 |
| L3 | 实例 / 轨迹 | 第 1 步 | 成本最低、时效最高 |
| L2 | 私有 / 策展 | 第 2 步 | 内部可信证据优先 |
| L1 | 公共 / 外部 | 第 3 步 | 成本最高、最后升级 |

L0 排第一，不是因为它"最重要"，而是因为它与 agent 自身的相关性最高——在你决定去找什么之前，你得先知道自己是谁、边界在哪。然后才是按任务相关性递减的 L3 → L2 → L1。

## 回头看那些工具，争论消失了

理解了分层之后，再把那些工具放回去，很多争论瞬间变得没有意义。

**Nocturne → L0。** 它做的是身份、原则、偏好、disclosure 条件唤醒、CRUD + 快照回滚 + 审计。它的价值不在于"更强的检索"，而在于——不再让人格和原则寄生在相似度检索上。Nocturne 唯一合理的位置就是 L0。

**lossless-claw → L3a。** 它的核心是持久化消息、DAG 摘要、context assembly。它的强项是"让旧上下文不被简单截断"。但它的代价也很明确：每轮参与 prompt 构建，prefix 稳定性下降，KV cache 复用变差。所以它是上下文引擎，不是通用记忆后端。

**memory-core / QMD / ClawDB → L3b。** 它们表面风格差异很大——memory-core 是默认的 Markdown + SQLite，QMD 是本地 sidecar 加 BM25 + 向量 + rerank，ClawDB 是服务化后端加 WAL/Parquet/DataFrame。但从 OpenClaw 的抽象层看，它们回答的是同一个问题：当 agent 调 `memory_search` 时，去哪查？

**Sirchmunk → L2。** Agentic Search，把检索当成 agent 行为序列而不是数据库查询。但它需要 LLM API（哪怕是本地端点），FAST 模式也不是零 LLM。如果输出不能稳定变成带引用的证据，它就不能进入事实链。所以它更适合做本地知识层的增益，而不是对话轨迹层的后端。

至于 **MemOS Cloud Plugin**？它的策略是每次运行前自动翻旧账——`before_agent_start` 注入 recall block，`agent_end` 写回。对某些产品化陪伴型 AI 完全合理。但如果你的目标是可预测、不吵、cache-friendly、默认不注入——那它就不该是默认路线。

## 什么才叫优雅

调研到这里，我开始形成一种直觉：好的记忆系统不是功能最多的那个，而是最克制的那个。

**默认不吵。** 不应该每轮都自动拉历史、自动塞 recall block、自动改写上下文。大多数时候 agent 不需要"回忆"，它只需要完成手头的事。

**默认可解释。** 每次 recall 都应该能回答三个问题——为什么现在查？查到了什么？证据在哪？如果答不上来，这次 recall 就不该发生。

**默认 cache-friendly。** 如果你的系统每轮都重写 prompt 结构，它就天然更贵、更慢、更难调试。这不是优化问题，是架构选择。

**默认可审计。** 尤其是 L0。约束是谁写进去的？何时改过？能不能回滚？如果人格层不可审计，你永远不知道 agent 为什么突然变了。

把这四条摆在一起，一个更优雅的默认策略其实很简单：

> L0 先决定约束；L3b 只在需要时被调用；L2 只在需要文档证据时被调用；L1 最后补外网。

不是每一层都要存在，不是每一层都要每轮参与。大部分时候，agent 只需要 L0 + 按需的 L3b 就够了。

---

## 不只是 OpenClaw 的问题

这套"先拆问题、再放工具"的方法，不只适用于 OpenClaw。

任何 agent 系统，只要遇到以下症状，都可以用同一个思路来厘清：

- 大家把"记忆"说成一个词，但实际在讨论三四件不同的事
- RAG、prompt cache、vectorDB、memory plugin、knowledge graph 被混在一起比较
- 工具越来越多，但行为越来越不可解释

先拆 Identity / Trace / Evidence / Web。再看每个工具接在哪个接口上。再决定默认策略是自动注入还是按需调用。

层位摆正，工具会互补。层位混乱，再强的工具也会互相污染。

---

## 你的第一步

最后给一个实操的选型指引：

- 你最在意**人格约束与审计** → 先做 L0（Nocturne）
- 你最在意**稳定、缓存、可预测** → L3b（memory-core / QMD / ClawDB）优先
- 你最在意**"丢文件就能搜"且接受 LLM 调用** → L2（Sirchmunk）做增益
- 你最在意**超长叙事连续性** → 再考虑 L3a（lossless-claw）作为特例

不要一次全装。从你最痛的那个问题开始，放对层位，剩下的以后再说。
