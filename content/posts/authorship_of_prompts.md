# 当 AI 越来越像基建，人类还通过什么留下自己？

_——Prompt 没有过时，它只是从工程退回了作者性_

## 魔法年代

在 context window 还不到十万、甚至不到一万的年代，prompt 的撰写已经像魔法一样了。那时还谈不上 engineering，context window 不够长，撑不起工程二字。

今天，领域已经从 Prompt Engineering 迁移到 Context Engineering，再到 Agent Harness。但对于如何写好一条 prompt，大众的认知似乎仍然停在 GPT-3.5 时代：人格设定、角色设定。主流研究迁移到了 Context Engineering，又在其能探索的方向逐渐收敛之后转向 Agent Harness。而 prompt 的技术越来越难以用一篇读完即明的文章来传递，相关领域的人多少有些曲高和寡。

二来，有一些问题，虽然精妙的 Prompt Engineering 能够解决，但一条粗糙的 prompt 配以更成熟的 Context Engineering 也能完成。如果把 Prompt 和 Context 看作游戏中的不同技能点，精妙的 prompt 技术能解决的复杂问题，粗糙的 prompt 配上足够的 context 常常也能达到类似的效果。增进 Context Engineering 和 Agent Harness 总归让人觉得在工程上更有前途，Prompt Engineering 的作用也就慢慢被淡化了。

## 两种精妙

我还记得第一次看到李继刚老师的「汉语新解」prompt 时的震撼——那是我这辈子绝对写不出来的东西。时至今日再看，这个断言并不完全成立，当然也并非完全不成立。

在我看来，这条 prompt 有两种精妙。

第一种是形式上的创新：与 Lisp 的结合。我当时甚至对 Lisp 没有概念上的理解，但它确实十分适合大语言模型。LLM 本身就像 Lisp 解释器，对它来说 prompt 既是语言数据，也是代码。将 Lisp 与 SVG 生成、与问题到答案的变换结合在一起，用同一种形式统一代码生成、SVG 排版与汉语释义的内容生成，再适合不过。这种形式上的创新，知道了便是知道了，很快就能模仿，倒也没什么门槛。

第二种精妙在于词汇的压缩：如何选择最精确的方法来传达，最犀利地提出问题，让大模型给出最犀利的答案。当时我不懂，而今我倒是有一些工程上的办法——让大模型阅读文本、总结规律，让它自己去选择那个最细腻的词语。还有笨方法：把一个想法跟大模型进行很多轮思辨性的讨论，让它把我难以用几句话准确表达的意思替我概括出来。虽然我在形式上仍然难以写出那样的框架，难以自己找到准确的几个词来表达脑中模糊的概念，但我知道怎么让 AI 帮我找到。

由人类自身选取方向再制作出来的 prompt，总归带有这个人几十年的独特生活和学习经验。正因为这些经验绝对不同于大模型平均的分布，它们能将模型的输出拉离默认的方向。

## 工程师的流派：宝玉

虽然这篇文章从李继刚老师的 prompt 开始，但我不想从他这一流派开始介绍。他的方法较为难以复制。我想从宝玉老师说起。他的方法更工程派一点，可复制、可迁移。由于他总是在尝试解决具体的问题，因此也积累了大量关于 AI 局限性的心得。

宝玉老师从 GPT-3.5 时代就十分活跃，一开始搬运大量国外 AI 资讯，逐渐将 AI 翻译作为一个不断进化的流程。那是 AI 还只有不到一万 token 上下文的时代，不够开发软件项目，但基本的信息翻译传达绰绰有余，尽管难以在文字风格上做到精细控制。

文章翻译看似简单，但如果要在风格上偏移 AI 的默认输出，就需要多步拆解。大家的方法不尽相同，核心思路是一致的：先逐字翻译，再进行风格转换；先确定大纲，再分部填充。流程如何拆解没有唯一的答案，因为一条 prompt 的输出不可被提前想象，对人类如此，对 AI 亦然。迭代就成了其中非常重要的一部分。时至今日，这些多步流程已经可以交给 AI 自行编排，而不必由人类手工分步调用。

除了基础的格式与风格决定，还可以要求 AI 参考原文素材进行写作风格的迁移。我曾读过一条用于分析写作风格的 prompt：

```
分析以下文本的：1. 句子节奏（长短句的分布规律）；2. 形容词密度；3. 情绪曲线。给出一份风格特征报告。
```

人类自己难以感受得到，但不同的人在写作时，确实有一种埋藏于文字、句子和段落之下的、呼吸一般的纹理节奏。我拿生活博主的文字试过，他们的节奏短小轻快。GPT 对我自己手写文本的分析则远比我的自我认知细致：句子节奏是"长长短，长长长短"的波形——多层递进拉长语义，偶尔一个短句收束总结；形容词不多，但偏好情感与认知并置的比喻性表达；情绪曲线则是以理驭情，有深层情绪纹理，但始终不失控。至于AI熟悉不过的著名的作家，还可以让它从思想深度等更多维度进行分析。

一个人的写作风格，其实有太多隐含的规则和维度，难以被人类自己肉眼解析。AI 虽然看得明白，但分析是一回事，执行时完美地遵循全部隐含规则又是另外一回事。若以信息传达保真作为标准，GPT-3.5 已经做得足够好；但若以全部不可明说的规则作为标准，在局部段落中保持全局一致性的人类写作，对 AI 来说实则比能够通过测试进行评审的软件工程更加困难。

## Skill：从流程到扁平组织

关于如何写好 skill，大多数方法所有人或早或晚都会发现——正如 Anthropic 在他们的工程博客中分享过的许多经验。

在我看来，skill 最伟大的创新之处，并不在于模块化，甚至不在于渐进式发现。这两点都极为显而易见。对于一些 AI 应该知晓但又并非总是需要被触发的工作流或领域知识，将它们以文档的方式进行存储，是再自然不过的事；对于用代码即可确认的流程，存作脚本而非自然语言，同样显而易见。真正的问题是：对于十几个甚至几十个毫无关联的、需要被渐进式发现的 prompt，应该如何组织，并且让 agent 能够自然、自发地去调用？

Skill 这个名字确实是一个非常好的概念。Anthropic 官方仓库给出的例子容易让人理解为它是一套工作流。宝玉老师的大多数 prompt 也落在这个范畴，将常用流程制作为 SOP。但我认为 skill 更深层的意义在于：它们共享一套格式，被扁平地用 meta 串起来，让 agent 能够自行组合编排。我想skill不是最终形态，但这一概念促使大家去动手将常用流程写下来，去与他人分享个人的知识，去思考一种任务的本质。

Skill 最初提出时，就附带了一个 skill-creator 的 skill。大家在打磨现有 skill 的过程中也很容易意识到，除了创作 skill 的 skill ，还需要能从实际使用中盘点经验、迭代打磨 skill 的skill。在 skill 这个概念出现以前，宝玉老师就已经一直在重复这个流程了，只不过一开始是手动调整，基于人类的建议而非 AI 的建议进行迭代。时至今日，大多数优秀的 skill 由 AI 创建、由 AI 优化，人类在这个过程中只提供最小的人类对齐。

Anthropic 的 skill 指南中提出的许多建议，其实与早年 Prompt Engineering 的技巧大致一致。如将具体执行细节留给 AI 决策，而非在工作流中进行过度约束；如在 few-shots 中只提及反例而不提出正例，以避免隐性约束。Anthropic 还提出了对 skill 进行测试和评测的方法，以避免其在自动优化过程中丢失原本想要支持的功能。

这些都是非常属于工程师的流派。对 agent 的经验来自于迭代，以及用工程方法进行控制和优化。

## 思想家的流派：李继刚老师

李继刚老师老师的 skills，若非套着卡片、slides 这些具体的产物形式，多数人恐怕很难将他公众号中的 prompt 视为一种 skill。他最近分享的 skills，无论是生成 card 还是知识图谱，都只是大家熟悉的知识产物形式。但李继刚老师的 skill 实际上并不需要一种具体的产物。他的 skill 不生产具体的东西，他的 skill 产生思想。思想再通过影响整个 session 来影响 agent 最终的产出，其实不需要在 skill 中定义它为哪一种产物服务。

从学习的方式来说：我们阅读一本书、看一部电影，需要通过一次次具体的观看去理解一种思想。我们很难被直接传授思想本身，那不是我们理解世界的方式。这就是李继刚老师的 prompt——它的 prompt 的正文是思想，prompt 本身不易被人类读者读懂，人类读者愿意阅读和能够理解的是 prompt 的产物。

## Prompt 的意义

我一直在思考：提示词除了提供 AI 尚且不了解的领域知识、公司的内部工作流程之外，还能提供什么？基于 AI 总是会比今天更加智能的假设，像李继刚老师这种风格的 prompt 的意义在哪里？

以写代码为例：至少对于今天的 Claude 或 GPT 来说，一条描述 coding 原则的 system prompt 不再那么重要了。哪怕你创造了一条极为精妙的 prompt，同样的模型，有它与没有它，给定合适的验证工具或外部评审，不同的 harness 都能到达类似的终点，差别仅仅在效率上。

那么由人来设计，或者引导 AI 设计一种 prompt，意义究竟在哪里？我其实是 Prompt Engineering 的拥趸。模型经过强化学习之后能达到更好的指令遵循，其实也可以通过 prompt 来诱导。效果当然远比不上强化学习，但它至少说明 Prompt Engineering 并非人类的一厢情愿，它确实可以塑造一个模型的外显特性。

我总是喜欢打比方：如果把模型的状态比作流水，把 harness 比作沟渠，prompt 上的调整就像建造不同的引导区——只要最低处是明确的（验证工具就是那个最低处），流水最终总会到达，区别只在河道的长短。

## 创作的是风格

我对李继刚老师的 prompt 还有另外一种想法：可以把它理解为画家的风格。你在观看一位画家的作品时，必须观看具体的作品才能感受到某一种主义。对于创作者来说，创造风格是困难的，但在同一种风格的框架下进行创作就是一种流程。莫奈可以画无数的印象主义的睡莲、印象主义的教堂——每幅各有不同，但对他来说就像呼吸一样容易。若是让他每一次画莲花时都要使用一种不同的风格，那创作便不再是流程，而是消耗灵感了。

当然，写一条 prompt 比发明一种绘画风格要简单许多。但同一个人在短期内能创造的原创 prompt，是极为有限的。

也许 AI 已经比人更加聪慧——这一点今天尚存争议，那么我们将这个命题放在 AI 比人类更聪明的某一天：人类写原创 prompt 的意义在哪里？我认为是风格。李继刚老师的汉语新解是一种风格，但作为观众，我们观看的并不是他的 prompt，而是这条 prompt 所生成的、对汉语词汇的另一种阐释。

过去的画家思考一种风格，然后在同一种风格下密集产出。莫奈在印象主义下画了几十年的睡莲与教堂；梵高在阿尔勒和圣雷米的两年间，以同一种旋涡般的笔触产出了数百幅作品。但达芬奇不同——他不停地在探索风格。达芬奇作为画家的作品并不多，甚至有很多未完成的作品，但他一生都在寻找。他的乐趣不来自于一幅作品，而来自于创造一种风格；当他为那种风格呈现出一幅作品之后，便又迅速转向下一种。

所以 AI 时代的创作，是被迫让每一个创作者都变成达芬奇。你要么像莫奈一样坚持一种风格，将它打造到 AI 的 context 能够满足的一致性之外；要么像达芬奇一样不断探索新的风格。或者像米开朗基罗一样，他的作品，无论是大卫、圣母、还是亚当，无论是绘画还是雕塑，风格上并不浓缩为某一种流派，但他始终在完美中表达永恒，在细腻中表达人类的脆弱性。我其实也不完全理解我自己说的这句话的涵义，但我感受得到：他的不同作品表达的是同一种精神。

也许米开朗基罗和莫奈并非两条不同的路——只是坚持的层次不同，一个在表层风格，一个在底层精神。

我想，人类在 AI 时代进行创作的意义，是风格。天才的存在并不阻碍普通人进行创作，同样，AI 的存在也不会阻碍普通人进行创作。我们除了寻找自己生活经验的共鸣者，也在寻找我们风格的受众。无论是以宝玉老师循环式迭代的形式，还是以李继刚老师压缩、思考、再压缩的形式，创作 prompt 能够表达风格。而这种风格一旦被创作出来，便可批量生产能被观看的作品。你用一条汉语新解 prompt 创造出来的一百条汉语解释，它们并不属于你；只有当你在原 prompt 的基础上融入了你自己的东西，它们才算是你的作品。

Prompt 是自然语言，但我觉得它就像代码。Prompt 之于 AI，是人类与瓶中精灵之间的适配程序。它把一个人的个体经验，用自然语言的形式，嫁接到伊甸园的果树上。在那枝上结出的果实，并非百分之百是人类的智慧，但对于每个人来说都是独一无二的。
