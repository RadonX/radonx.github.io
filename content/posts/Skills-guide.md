# Agent Skills：从团队共享到个人工作流的实践探索

## 引言

2026 年初，AI Agent 生态正在经历一场深刻变革。以 Anthropic 的 Claude Code、Skills 以及新近爆火的 OpenClaw为代表的工具链，正在重新定义开发者与 AI 的协作方式。本文基于一场多位一线实践者参与的分享会，系统梳理了 Skills 的开发实践、自动化工作流构建、安全治理、评估方法论以及行业趋势判断。

---

## 一、Skills 是什么，为什么重要

Skills 是 Anthropic 围绕 Claude Code 推出的一种上下文优化机制——本质上是用自然语言编写的指令文件（如 `skill.md`），用于告诉 AI 在执行任务时应遵循什么规则、使用什么数据、调用哪些工具。

Vercel 平台负责人 Andrew 将 Skills 定位为让模型保持最新知识的关键手段。以 Next.js 为例，模型的预训练数据往往停留在旧版本（如 Next.js 14），而实际框架已经迭代到 15、16，人体工学（ergonomics）发生巨大变化。Skills 提供了一种机制，让开发者能将最新的最佳实践注入到 AI 工作流中，弥补预训练数据的滞后。

Andrew 强调，Vercel 对 Skills 的理解更接近互联网的开放模型，而非 App Store 的封闭模式——每个人都可以发布自己的 Skill，其价值由社区大众评判，而非由某个中心化审核机构决定。Skills.sh 排行榜展示的技能全部来自真实开发者运行 `npx skills-app` 发布，而不是从 GitHub 抓取。

## 二、从团队共享到个人工作流：Skills 的边界与评估

### 1. 为什么不能“把所有 Skills 分享给团队”

分享者从两个角度提醒：**并非所有 Skills 都适合团队共享。**

1. **通用性 vs 专用性**  
   Skills 天然处于“泛用”和“精准”的光谱之间：
   - 越泛用，就越不针对具体场景，输出更模糊；
   - 越锐利，就越专属于某个具体领域和上下文，难以迁移到其他团队。

   这与传统 deterministic library 不同：后者可以在不破坏原有功能的前提下不断扩展；而 Skills 一味往“更强大”演化，往往等价于“更泛用、但更不精确”。不同团队实际上需要的是在各自领域拥有不同锐度（sharpness）的 Skills，而不是一个巨大的“公司级通用 Skill 库”。

2. **评价标准 vs 生产环境的隔离**  
   分享者借用了机器学习中数据切分的比喻：training / validation / test 必须隔离。如果你把 test set 也拿去训练，模型在指标上会很好看，但并不泛化。

   对 Skills 来说，如果你把用来评估代码质量的“评价 Skill”和生产环境中实际写代码用的 Skill 打包共享，相当于把 test set 混进 training set——会导致“开卷考试”：所有人都知道怎么写出“看上去符合评分标准”的结果，却不一定符合真实业务场景。

   结论是：**评价标准与生产 Skill 需要保持适度隔离，且独立迭代。**

### 2. Vercel 的实证方法论：Eval 驱动的 Agentic Experience

Vercel 工程师 Jude 从“Agentic Experience”的角度分享了他们在 Next.js 生态中的探索。

#### 2.1 从 MCP 到 Skills：技术不等于解决问题

起点是一个所有框架都会遇到的“知识窗口期问题”：

- 模型预训练时只见过 Next.js 14；
- 实际用户可能在使用旧版本，也可能在用最新的 15 / 16；
- 结果是：模型生成的代码要么过旧、不兼容，要么“过新”，与当前项目版本不符。

他们最早尝试的是 MCP，将 Next.js 的 dev server 同时开放一个 MCP 端口，让 Agent 像 pair programmer 一样连上来操作。但实践发现，虽然技术方案“很酷”，却增加了大量 overhead，并没有真正解决开发者的痛点：很多时候“复制报错信息到 ChatGPT”比接一堆 MCP 更简单。

#### 2.2 Skills 的 A/B 测试：加了不等于有灵魂

随后团队转向 Skills，设计了 20 个固定的编程任务，对比“有 Skills”和“无 Skills”的表现。提示词保持中性，不刻意提示模型“使用某某 Skill”。

评估结果出人意料：**加上 Skills 后，通过率几乎没有变化。** 即便在提示中反复强调“要利用这些 Skills”，提升也非常有限。

这个结果打破了一个常见幻觉：**给 Agent 安装 Skills，不等于给它“注入灵魂”。**

#### 2.3 真正有效的方案：8KB 压缩文档注入 system prompt

在与 Claude 的反复讨论中，Jude 得到一个关键建议：直接将当前版本 Next.js 的文档高度压缩成约 8KB 的摘要，以一行配置的方式注入到 `.claude.md` 的 system prompt 中。

实践证明，这个看似“粗暴简单”的方法，在解决知识窗口期问题上效果非常显著，远好于仅依赖 Skills 选择或 MCP 连接。

#### 2.4 Eval–假设–迭代的方法论

Jude 将这套探索总结为一条可复用路径：

1. **从具体问题出发**：不要从“我手上有 MCP / Skills / Agents”出发，而是从“开发者到底遇到了什么问题”出发。
2. **先建 Eval**：哪怕只有 40–50 个测试用例，也足以提供方向性指导。
3. **用 Eval 验证假设**：不断尝试 MCP、Skills、压缩文档等不同方案，理解模型在不同方案下的行为。
4. **找到通过 Eval 的最简方案**：可能是某种组合，也可能只是“一行 system prompt”。

## 三、实践者视角：多角色、多领域的 Skills 落地

### 1. 创意工作流：向阳乔木（产品经理 & 内容创作者）

向阳乔木（向阳老师）有多年产品经理经验，曾在美团工作，目前一边创业一边分享 AI 技巧。他的路径很典型：

- **初始阶段**：看到 Skills 需要写 `skill.md`、编排规则、甚至连接 MCP，觉得门槛很高；
- **实践后发现**：难度远低于预期，且灵活性极强——只要用自然语言告诉 Claude “帮我写一个 CloudCode Tool，完成某事”，很多样板代码可以自动生成。

#### 1.1 一句话论文解读流水线

他的论文解读 Skill 工作流大致是：

1. 根据链接下载论文 PDF；
2. 用专门工具（如 Marker）解析出文本和图表；
3. 让 Claude 结合个人写作风格生成图文并茂的解读文章；
4. 调用 Gemini 等模型按章节自动配图；
5. 通过 Obsidian 插件或浏览器自动化，一键发布到公众号或 X。

用户侧体验极其简单：输入“读一下这篇论文并写篇面向大众的解读文章”，剩下全部自动完成。

#### 1.2 一句话生成 MV

另一个更极致的例子是“一句话生成 MV”的 Skill 串联：

- 用 Suno API 生成音乐和歌词；
- 用 Whisper 把音频转为带时间轴的字幕；
- 将原始歌词与识别结果比对，修正时间轴；
- 调用图像生成模型，为每句歌词生成对应画面；
- 用脚本将图片、字幕和音频合成为完整 MV。

指令可以随意自然，例如：“点一首关于 X 的冲浪摇滚乐，做成 MV。”背后是多个 Skills 串联完成复杂的多模态流水线。

#### 1.3 自然语言迭代：从页面级到工作流级的“泡泡编程”

当某个环节效果不理想时，他并不是去改代码，而是直接在 Claude 里用自然语言说出修改意见，让 AI 自动重写 Skill 配置和脚本。这种方式类似早期的“Bubble Coding”（可视化页面编排），但粒度提升到了“工作流级”——你在对话框里改的是整条业务链路，而不是某个页面组件。

#### 1.4 商业价值与“不再无条件分享”

向阳老师坦言，以前写 Prompt 会毫无保留地公开，但现在对完整 Skills 的开放开始犹豫，因为：

- 一个 Skill 封装了他的多年经验、审美与流程；
- 只要拿到这个 Skill，别人几乎可以 1:1 复刻他的生产力。

这也是他判断 Skills 具备商业价值的信号：**当你开始犹豫要不要分享，说明它真的值钱了。**他鼓励大家尽早用 Skills 把自己的经验沉淀下来。

> 注：文中“向阳”“桥木无常”均指同一位分享者“向阳乔木”。

### 2. Rust 开发者的工程实践：Alex

Rust 开发者 Alex 从工程团队协作与规范的角度切入 Skills。

他的工作包括：

- **领域 Skill 封装**：将某个 Rust 跨平台 UI 框架的语法、组件体系、最佳实践封装成 Skill，供团队统一使用。
- **热加载 + 自动注入**：利用 Skills 的 hooks，在对话不同阶段（发消息前、规划结束后等）自动注入领域知识，引导 Agent：
  - 使用正确的依赖版本；
  - 遵守团队安全规范（如 `unsafe` 使用策略）；
  - 维持统一的代码风格。
- **SkillWork 包管理**：为团队 Skills 提供依赖管理、供应链审查、自动化测试等能力，支持本地与全局配置。

他观察到，自己在 2025 年后已经“基本不再手写代码”，几乎所有代码都由 Agent 生成，但质量参差不齐。Skills 在这个过程中逐渐从“知识图谱”角色演变为“AI 引导工具”：重点不在于把知识堆给模型，而在于为它设定清晰的决策边界。

### 3. 写作自动化与自我优化：陈峰

陈峰专注内容创作和自媒体增长，他的核心创新在于：**让 Skills 自己学会变好。**

- 将写作拆成多个环节：选题、结构设计、正文撰写、标题优化、平台适配等，每一步一个独立 Skill，以“01-选题、02-大纲、03-正文”这样的编号组织。
- 建立“反馈–改进”闭环：
  1. AI 生成内容；
  2. 人工评估质量并发布；
  3. 收集平台反馈（阅读量、转化等）；
  4. 用专门的“自更新 Skill”把反馈交给 AI，总结规律；
  5. 由该 Skill 直接修改原有 Skills 配置。

经过数月迭代，他的 Skill 体系已经高度个性化，不仅匹配他的写作风格，也适配目标受众的阅读偏好。他正在进一步探索：如何在同一个 Skill 中，区分“创作者设定”（写作哲学）和“用户信息”（具体账号、受众画像），从而既能复用核心方法，又不丢失个体差异。

## 四、OpenClaw：从 Skills 到全自动化个人助理

雨晨则从“Agentic Experience + Skills”迈向了“长期运行的个人 Agent 系统”，代表项目是近期在推特上爆火的 OpenClaw。

### 1. 架构与部署

- 运行在个人 Mac Mini 上，通过 Telegram 进行远程操控；
- 通过物理隔离保障安全（生产环境与日常电脑隔离）；
- 使用多个模型供应商的 API（GPT-4.5、Claude Sonnet 4.5 等），并通过 OpenClaw 框架统一调度。

### 2. 项目管理自动化

他接入了 Linear API，让 OpenClaw 充当“自动化项目经理”：

- 每天早上生成项目进度简报；
- 监控 Sprint 中的 ticket 状态；
- 若某个 ticket 两天无更新，自动 @ 对应 assignee 催进度；
- 后续计划接入更多协作工具，实现跨平台的工作流协调。

### 3. 双 Bot 协作与自监督

雨晨部署了两个角色不同的 Bot：

- **Crabby**：面向外部社区，权限和访问范围受限；
- **Crawly**：个人助理，权限更宽，负责复杂自动化操作。

他甚至会定期让 Crawly review Crabby 的工作质量，对输出提出修改建议，自动更新相关 Skills 并重新部署，实现一种“Agent 自监督+自进化”的模式。

### 4. 内容发布与链路打通

在内容发布方面，他正在打通“Telegram → AI 写作 → X 发布”的全链路，让日常在 Telegram 中发的一句想法，能够自动走完整的整理、扩写、配图、排版与发布流程。

## 五、Token 成本与模型选择：先看天花板，再谈省钱

围绕模型成本，现场也有大量讨论。

- 雨晨的策略是：**现阶段无脑用最好的模型**（GPT-4.5、Claude Sonnet 4.5），优先看到“未来的天花板”——先证明工作流在高质量模型下的上限，再考虑向便宜模型迁移。
- 另一位分享者提到：团队在密集使用 Claude + Superpowers + Web 搜索后，个人额度一天就能消耗 75%，团队负担较重，只能将结构化数据处理等“简单任务”分流到开源或免费模型（如 Whisper、Apple Foundation Model）。

现场形成的共识是：**现在更像是“流量还很贵的 3G 时代”**。需要一边用最好的“流量”打磨产品，一边为未来 Token 成本大幅下降提前布局。率先把“水管”接到自己业务上的人，将在 Token 变得廉价时获得巨大的规模优势。

## 六、Skills 的工程化管理：UNIX 哲学的回归

主持人从 UNIX 哲学的角度总结 Skills 的设计理念，认为 Anthropic 的 CloudCode 团队里一定有资深 UNIX 黑客：

- **单一职责原则**：每个 Skill 只做好一件事。实践路径往往是：从“大而全 Skill”起步，逐步拆分为“小而专 Skill”。
- **管道式组合**：多个小 Skills 串联，组成复杂工作流。比如“总结 Skill + 写作风格 Skill + 视频剪辑 Skill + 图片生成 Skill”，可以完成从抓取 YouTube 视频到多平台分发的流程。
- **文本流处理**：所有交互围绕文本展开，自然跨平台、跨语言、跨时代。
- **Shell 一等公民**：CloudCode 让“自然语言 → Shell 命令”的映射异常顺滑，非工程师也能间接操控底层系统。

在实际管理上，他们总结出三层结构：

| 层级   | 适用范围             | 示例                          |
|--------|----------------------|-------------------------------|
| 全局   | 所有场景通用         | 通用工具类 Skills            |
| 项目   | 特定项目 / 业务领域  | 某产品线的 API/规范 Skill    |
| 个人   | 个人偏好与习惯       | 写作风格、审美、文件命名规则 |

有分享者用 GitHub Actions + 开源模型自动抓取、分类、去重 Skills，再通过 Symlink 分发到不同项目目录，实现了“统一维护、按需挂载”的工程化管理。

## 七、安全治理：不可忽视的底线

多位实践者强调：**Skills 的供应链安全必须严肃对待。**

- Skills 用自然语言描述行为，恶意逻辑往往更隐蔽，例如看似无害的“下载某个文件并执行初始化脚本”，实际上可能具备破坏性。
- Vercel 的治理思路包括：
  - Skills.sh 上每个 Skill 都绑定到 GitHub 仓库，必须拥有该仓库才能发布；
  - 平台对仓库内容进行持续扫描；
  - 依赖关系有意保持“扁平”，避免出现类似 NPM 深层依赖树难以审计的问题。

实践建议：

- 优先使用 Skills.sh 等具备一定审查机制的平台；
- 对未知来源的 Skill 保持怀疑，在沙盒环境中测试；
- 对涉及账号、资金、生产环境的操作采用物理或网络隔离。

## 八、行业格局：Anthropic 作为标准制定者

从更宏观的角度看，Anthropic 在 Agent 与 Skills 生态中，已经形成了“定标准”的姿态：

- **Constitutional AI**：为可治理、可约束的 Agent 奠定基础；
- **Sub-Agent 架构**：提供多 Agent 协作范式；
- **MCP（Model Context Protocol）**：成为工具 / API 接入的事实标准，被 Cursor、VS Code 等主流平台跟进；
- **Skills**：成为上下文优化与工作流编排的新标准，发布不久 OpenAI 及其他厂商便快速跟进。

有人用一句话概括：“三流公司做产品，二流公司做服务，一流公司定标准——从这个角度看，Anthropic 已经是一流公司了。”

当然，围绕 OAuth 封锁、地区管制等也存在争议，但在 Agent 与 Skills 形态的创新速度和深度上，Anthropic 目前仍然是行业最前列的玩家之一。

## 九、结语：你的经验，就是你最有价值的 Skill

这场分享会贯穿始终的主线是：**Skills 不仅仅是一种技术机制，而是一种新的生产力范式。**

- 它让个人经验、专业知识、审美判断被编码为可复用、可组合、可迭代的工作流；
- 它让 AI 从泛化的“通用助手”，转变成深度理解你领域和风格的“专属协作者”；
- 它将“人 + AI”的关系从“问答式使用”，推向“长期协作关系”和“自动化生态”。

向阳乔木说：“你可能花三五天写好一个 Skill，但以后再做这件事，不管用什么工具，它都可以是 AI 的一部分。”而 Jude 的方法论提醒我们：从问题出发，用 Eval 验证，用最简方案落地。

无论你是工程师、产品经理还是内容创作者，现在都是构建“个人 Skills 资产”的最好时机。等到 Token 真正便宜、Agent 真正普及的时候，那些已经把自己经验系统化为 Skills 的人，将天然站在新的生产力曲线前端。
