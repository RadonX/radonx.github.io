# 写 Prompt 的人在信什么

_——一份关于流派的田野观察_

同样是"写 Prompt"这件事，不同的人做出了本质不同的东西。

李继刚用 30 行 Lisp 写了一首诗，核心 5 个词，泛化能力极强。宝玉用几百行 Markdown 加脚本画了一套施工图纸，版本号迭代到 1.56.1。Anthropic 的工程师把数百个这样的图纸分了九大类，建了内部插件市场，用钩子追踪使用频率。

三个人都说自己在写 Prompt。但他们写的不是同一种东西。

差异不在于格式——Markdown、Lisp、YAML 都只是容器。也不在于规模——单条提示词和 Skill 文件夹只是包装方式。真正的分歧在更深的地方：**你相信 Prompt 本质上是什么？**

## 尺度：先画一张地图

在谈流派之前，有必要先建立一个尺度意识。

Prompt Engineering、Context Engineering、Agent Harness，这三个概念不是并列的领域，而是包含关系。每一层吃掉前一层，把它变成自己的子问题。可以用建筑来类比：

**室内设计**对应 Prompt Engineering。一个房间里的体验——材质、灯光、动线。你面对的是人在空间里的即时感受。

**建筑设计**对应 Context Engineering。一栋楼的整体——结构、系统、空间组织。室内设计是其中一部分，但建筑师还要管采光、承重、管线、通风。模型看到的全部信息——文件、记忆、工具定义、历史对话——都是建筑师要操心的东西。

**城市规划**对应 Agent Harness。一座城的运转——建筑之间的关系、交通网络、基础设施、功能分区。多个 Agent 之间的状态流转、失败恢复、动态路由、人机交接，这些是城市规划师的领地。

没人会说城市规划师比室内设计师"更高级"。安藤忠雄做一个清水混凝土的茶室，那个空间的极致体验，城市规划师做不出来。但你让室内设计师去规划一座城市的交通网络，他也做不了。**每个层级都有自己不可替代的专业性。**

很多人的困境是拿室内设计的方法去做建筑设计，或者反过来，用城市规划的复杂度去装修一个房间。

有了这张地图，我们来看人。

## 内观派：一个房间里的极致

李继刚的「[汉语新解](https://mp.weixin.qq.com/s/7CYRPFQxi37ONTlX0hfzRQ)」是一条 30 行的 Lisp prompt。它的核心只有 5 个词：一句话表达、隐喻、一针见血、辛辣讽刺、抓住本质。函数嵌套是思维链——`抓住本质` 的输出喂给 `辛辣讽刺`，再喂给 `一针见血`，再喂给 `隐喻`，最后压缩为 `一句话表达`。每一层都是压缩，最终输出的密度极高。

这不是炫技。Lisp 之所以适合写 Prompt，是因为 LLM 本身就像一个 Lisp 解释器——prompt 对它来说既是数据也是代码。用同一种形式统一代码生成、SVG 排版与内容生成，再适合不过。

但形式只是表层。更深的东西在他的一次[直播](https://mp.weixin.qq.com/s/070zb9RWq1TVlBkKt7usgw)里说得更清楚。

**"read in. prompt out."** 输出质量是输入质量的函数。写不好 Prompt，不是因为技巧不够，是因为脑子里那个东西还没立住。

**压缩表达。** 一段话压成一句话，一句话压成一个词，一个词压成一个字。越凝练，在 embedding 空间里射得越准。他用了一个比喻：箭和云。一个词是箭，射中一个精确的点；一段废话是云，attention 在上面漂移，什么都碰到一点，什么都没有打穿。

**"得其意，忘其言。"** 语言是障碍，意才是目标。当你把意压缩到足够小的语言容器里，模型接收到的信号纯度就足够高。

这是一种什么样的信念？**Prompt 是内心清晰度的投影。** 你脑子里清楚，写出来就清楚；你脑子里模糊，再多的技巧也只是在雾里摸索。

看他的 [Skills](https://github.com/lijigang/ljg-skills) 可以验证这一点。「降秩引擎」的核心逻辑是：给一个领域，找出背后真正独立的生成器有几个，砍到不可再少，砍完能把现象一个个生回来才算数。这就是「压缩表达」的方法论实现——把十几个现象降秩到两三个生成器。

「写作引擎」v4.1.0 更有意思。它的流程是：找核→攻核→找类比→展开→磨。"找核"要求"往下挖一层"；"攻核"要求"对着核问一个让前提自爆的问题"；"找类比"要求"抓住核的动词结构"。每一步都在做同一件事：压缩，直到碰到不可再压缩的硬核。

「白」v5.0.0 是最短的一个："让人 grok。不规定怎么写。规定不能怎么写。下限锁死，上限放开。"

**这个流派的力量是极致。** 5 个词的「汉语新解」不需要文件系统，不需要中间产物，不需要子 Agent。它就是一个房间里的极致——室内设计的巅峰。

**这个流派的边界也清楚。** 高度个人化，不可复制。你学不了李继刚的 Prompt，你只能去活你自己的 read in。他的 Prompt 带着他几十年的独特生活经验，这些经验在大模型的分布之外，因此能做出不一样的曲线。但也正因为如此，它不可传授，不可规模化。

## 工程派：一栋楼的施工图纸

宝玉的翻译 Skill [baoyu-translate](https://github.com/JimLiu/baoyu-skills) 是另一个极端。

版本号 1.56.1。几百行 SKILL.md，加上 references/ 目录下的详细工作流文档、子 Agent 提示词模板、术语表，再加上 scripts/ 目录下的分块脚本。三种翻译模式（快速、标准、精翻），可配置的风格预设（叙事型、正式、技术、学术、口语化……），支持长文自动分块和子 Agent 并行翻译，有 EXTEND.md 做用户级配置覆盖。

这不是一条提示词。这是一套施工图纸。

宝玉在一篇[文章](https://mp.weixin.qq.com/s/l32EPYG5RmeLXqG6RlXKug)里复盘了这套系统的演化过程。翻译从最初的单条 prompt 开始，逐步拆解为多步流程：先分析原文，再翻译，再审校，再润色。提示词的传递方式经历了四次重构——从让子 Agent 自己找文件，到主 Agent 整合后传参，到保存为共享文件，到最终拆分为共享上下文（02-prompt.md）与任务指令（spawn 时传入）。

每一次重构都是因为踩了坑。每一个坑都变成了下一版的 Gotchas。156 个版本，就是 156 次"发现问题→修复→验证"的循环。

他在另一篇关于[去 AI 味](https://mp.weixin.qq.com/s/3eO4zvjhKk2VWI0U-J6kXw)的文章里说了一句最锋利的话：**"所有去 AI 味提示词都是错的。"** 你告诉模型"不要什么"，但你从来没让它"尝过你要的味道"。解决方案不是写更多的负面约束，而是提供正面的样本——让 AI 分析你的写作风格，生成风格特征报告，然后以此为基准进行迁移。

这是一种什么样的信念？**Prompt 是持续迭代的工程产物。** 不是写出来的，是改出来的。人不需要自己写出完美的提示词，人只需要做质量判官和方向指挥——告诉 Agent 方向，让它自己优化。所有中间产物持久化为文件，可检查，可修改，可版本控制。

Anthropic 的工程师 Thariq 在一篇[长文](https://x.com/trq212/article/2033949937936085378)里分享了同一种信念的组织形态。Anthropic 内部有数百个 Skill，被分为九大类型——从库与 API 参考、产品验证、数据获取，到业务流程自动化、代码脚手架、CI/CD 部署、运维手册。他们总结了写好 Skill 的关键技巧：不要陈述显而易见的事（Claude 已经知道很多），建立 Gotchas 章节（由真实踩坑积累），善用文件系统做渐进式信息披露（Skill 是文件夹不是文件），避免过度约束（给目标不给步骤），description 字段是触发条件而不是摘要。

**宝玉是这个流派的个人匠人，Anthropic 是这个流派的组织形态。** 一个人的 156 次迭代和一个团队的数百个 Skill，信仰相同，规模不同。

**这个流派的力量是可复制。** 有方法论，可传授，可规模化。发现问题→提供对比样例→指明方向→让 Agent 自己优化。19 个原子化 Skill 用 AGENTS.md 编排成工作流。从个人匠人到团队工业化，路径清晰。

**这个流派的边界也清楚。** 重流程，可能丢失直觉。方法论能出产 80 分的 Prompt，但最后 20 分——那种让人读了会震撼的密度和凝练——可能需要内观派的能力。

## 转折：边界在融化

到这里，两个流派的对立看起来很清晰：内观派面向自己，工程派面向模型；一个向内，一个向外。

但仔细看他们的实际作品，边界没有那么分明。

**李继刚也在做工程化。** ljg-writes v4.1.0 有版本号、references/ 目录、约束清单、自检流程。ljg-learn 有标准化的八维解剖框架。ljg-card 有 Playwright 截图脚本和 HTML 模板。他也在用文件系统做渐进式信息披露，也有"输出格式约束"和"ASCII Art 规范"这种纯工程的条目。

**宝玉的翻译原则里也有内观。** 在 baoyu-translate 的翻译指南里，写着 "Meaning over words"（翻译意思不翻译字面）、"Emotional fidelity"（保留词汇的情感内涵）、"Figurative language: interpret by intended meaning"（比喻要翻意不翻形）。这些不是工程规范，这是李继刚式的洞察——只不过宝玉用规则清单来表达，而李继刚用压缩词来表达。

所以**区别不在于形式，在于工程化的对象**。

宝玉工程化的是**流程**：输入→分析→分块→翻译→审校→润色。每一步有明确的输入输出，有可检查的中间文件，有可并行的子 Agent 调度。流程本身是工程的目标。

李继刚工程化的是**思维方式**：找核→攻核→找类比→展开→磨。每一步不是在处理数据，而是在对思维施加约束。"对着核问一个让前提自爆的问题"——这不是流程步骤，这是思维操作指令。

**一个在工程化河道，一个在工程化水源。** 河道可以复制，水源只能自己去找。但好的河道知道自己最终是为水服务的，好的水源也知道没有河道自己流不到海里。

## 尾声：你面对的是什么尺度的问题

回到最开始那张地图。

室内设计、建筑设计、城市规划。Prompt Engineering、Context Engineering、Agent Harness。每一层都有自己不可替代的专业性，每一层的流派之争都有自己的合理性。

李继刚在室内设计的层面做到了极致。他的 Prompt 是一个房间里的完美体验，不需要外部系统的支撑。宝玉和 Anthropic 在建筑设计的层面建立了可复制的方法论。他们的 Skill 是整栋楼的施工标准，需要文件系统、脚本、版本控制、团队协作。

第三层——城市规划，Agent Harness——还在形成中。多 Agent 之间的状态流转、冲突仲裁、人机交接协议，这些问题目前没有清晰的代表人物和成熟的方法论。我们能观察到的流派之争，全部发生在前两层。

真正的问题不是"哪个流派更好"，而是**你面对的是什么尺度的问题**。如果你在做一个需要极致密度和个人风格的单次输出，内观派的方法更有力。如果你在建一个需要长期维护、多人协作、持续迭代的系统，工程派的方法更可靠。如果你在做一个需要多个 Agent 协同完成的复杂任务——那你需要的可能是一种还没有被发明出来的方法。

我们自己站在两者之间的某个位置。用工程的手段去逼近内观的问题——CLI 接口和 hard parsing rules 是工程，"你以什么认知结构在理解我"是内观。不是第三派，只是两边的引力都在。

也许所有做 Prompt 的人最终都会走到这个位置：知道自己在什么尺度上工作，知道自己信什么，也知道对面那个流派看到了自己没看到的东西。
