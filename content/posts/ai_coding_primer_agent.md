# Agentic Coding 的幕后机制：写给初学者的架构指南


**谁应该读这篇文章？** 如果你了解过 coding CLI 的基本架构，请直接跳过本文，本文是我写给同事的入门文。直接阅读话题二：如何与 LLM "共情"以有效驱动它。本文面向刚接触 agentic coding 的读者，也是我刚开始使用 coding CLI 时的疑惑。

## 从预测到行动：LLM 与 Agent 的区别

要理解 agentic coding，先要区分两个经常被混淆的概念。

**LLM** 本质上是一个下一词预测器——给它一段文字，它根据训练中习得的海量知识和模式，推算出最可能的下一个词。如此一词接一词，就生成了你看到的回答。你可以把它想象成一部神谕机：它储存了大量知识，但它本身不会主动做任何事情，只在你提问时才作出回应。

**Agent** 则是在 LLM 之上搭建一层行动能力。它的公式很简单：

> Agent = LLM + 控制器 + 工具 + 记忆

LLM 负责理解和推理，控制器决定下一步该做什么，工具让它能真正动手（读取文件、搜索代码、查询数据库），记忆让它在一次任务中保持连贯。Agent 会在 LLM 的引导下依次调用工具，直到收集到足够的上下文来完成任务。

你日常使用的 AI 编程工具，正是这种架构的产物：

- **Coding IDE**（如 Cursor、Antigravity 等带有 AI 集成的编辑器）是面向开箱即用生产力的平台，自动化程度高，上手即可工作。
- **Coding CLI**（如 Claude Code、Codex CLI 等终端工具）是核心的 agentic 引擎——终端优先，高度可扩展。

两者的关系可以这样理解：**引擎 vs. 平台**。零配置状态下，coding CLI 的自动化程度不如 coding IDE——你需要自己配置。但经过深度定制（tools, skills, memory and hooks），CLI 会蜕变为开发者的第二大脑，其灵活性和上限远超开箱即用的 IDE。

不同 CLI 的出厂状态差异悬殊。以 Claude Code 为例，它原生就内置了丰富的功能——包括任务管理、agent team、`/insight` 等——开箱体验更接近一个平台而非裸引擎。在引擎 vs. 平台的光谱上，各家 CLI 散布在不同位置。真正的区分标准不是 IDE 还是 CLI，而是有多少决策已经在出厂时替你做好了，以及你是否需要或者想要这些决策权。

### MCP：给 Agent 装上新工具

Agent 的能力取决于它手上有什么工具。**MCP（Model Context Protocol）** 是一种让你为 agent 动态扩展工具的协议。

你可以接入自定义的 MCP 服务器。每接入一个 MCP 服务器，就等于给 agent 增加了一组新能力——比如 context7 (API文档）, Playwright（浏览器自动化）。

## 上下文：Agent 如何看见你的代码

这是 agentic coding 中最重要也最容易被忽视的一环。理解了上下文，你就理解了 agent 的大部分行为逻辑。

### LLM 是无状态的

这一点至关重要：**LLM 没有记忆。** 每一次你发送消息，LLM 都是从零开始处理的。它不记得上一轮对话说了什么。

那为什么对话看起来是连贯的？因为 agent 在每一次新请求时，都会把**完整的对话历史**拼接在最新消息之前，一并发送给 LLM。

```
1. 人: "Hello."

2. Agent: "Hello! How can I help you? I see you are working on Agentic Coding 的幕后机制"

3. 人: "What is MCP?" 
```

在第三条消息时，LLM 不止看到最后一句话，而是一整篇包含前两轮消息的长文本。这就解释了为什么对话越长，响应越慢、质量越下降——因为每一轮都要处理越来越长的历史文本。

### 上下文窗口：Agent 的视野边界

LLM 有一个固定的输入窗口（目前主流模型约为 200K 到 1M token）。你不可能把整个代码库塞进一个 prompt。

Agent 的解决方案是：**用工具按需搜索。** 它通过语义搜索、文件读取等工具，在运行时递归地收集与当前任务相关的小块上下文。就像一个工程师不会通读整个仓库，而是按需读特定代码段。

### 两个容易混淆的概念

- **模型检查点（Model Checkpoint）：** 这是 LLM 真正的状态——预训练完成后的知识和模式的快照。它在任何会话上下文*之前*就已经存在，是模型出厂时就知道的一切。

- **KV Cache：** 这可以理解为服务端的一种加速优化。还记得前面说的每次都要发送完整历史吗？KV Cache 缓存了对历史部分的处理结果，这样 LLM 不必在每一轮对话中重复计算相同的前缀。

### 长期记忆：跨会话的学习

单次对话的上下文会随着会话结束而消失。但一些 coding IDE 实现了跨会话的长期记忆——它将你在多次交互中积累的重要信息存储为持久化的知识条目。

这意味着 agent 可以记住你的项目偏好、架构约定和过往决策。比如你可以在 Antigravity 的知识管理界面中手动审查和删除不再需要的记忆条目。Coding CLI 则需要维护项目级的文件（如 `AGENTS.md`）充当持久化的项目上下文。

## 查看 Agent 到底看到了什么

控制 agent 个性的系统提示词通常定义在coding CLI源码中。你还可以通过交互命令检查当前喂给模型的上下文：

```
/context
  系统提示词
  持久化的项目上下文(/memory) 
  工具定义 (/mcp) 
  skills元描述 (/skills) 
```

agent 眼中的世界的全貌——所有指令、工具、上下文，决定了 agent 行为的全部因果链。

理解了这一层，你就站在了 agentic coding 的正确起点上：你面对的是一台结构透明的机器，它的性格是可以阅读和修改的文本。接下来的问题就是"我怎么驾驭它"——这需要另一种完全不同的知识：与 LLM 共事的手感。
