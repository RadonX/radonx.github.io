# Prompt 的三种信仰：一篇关于流派的观察

## 结构稿 v0.2

---

## 文章主旨

Prompt 写作已经分化出了不同的流派。这些流派之间的差异不在于格式（Markdown vs Lisp vs YAML）、不在于容器（单条提示词 vs Skill 文件夹），而在于一个更根本的分歧：**你相信 Prompt 本质上是什么？**

这篇文章不是技巧清单，是一份田野观察。

---

## 概念框架：三层包含关系

在讨论流派之前，先建立一个尺度意识。Prompt Engineering、Context Engineering、Agent Harness 不是并列的三个领域，而是包含关系——每一层吃掉前一层，把它变成自己的子问题：

```
Agent Harness (城市规划)
  └── Context Engineering (建筑设计)
        └── Prompt Engineering (室内设计)
```

类比建筑领域：

- **室内设计（Prompt Engineering）**：一个房间里的体验——材质、灯光、动线。面对的是人在空间里的即时感受
- **建筑设计（Context Engineering）**：一栋楼的整体——结构、系统、空间组织。室内设计是其中一部分，但建筑师还要管采光、承重、管线、通风
- **城市规划（Agent Harness）**：一座城的运转——建筑之间的关系、交通网络、基础设施、功能分区。每栋楼的设计是其中一个节点

关键洞察：**每个层级都有自己不可替代的专业性。** 没人会说城市规划师比室内设计师"更高级"——安藤忠雄做一个清水混凝土的茶室，那个空间的极致体验，城市规划师做不出来。反过来，你让室内设计师去规划一座城市的交通网络，他也做不了。

很多人的困境是**拿室内设计的方法去做建筑设计**，或者反过来，**用城市规划的复杂度去装修一个房间**。

目前我们观察到的流派之争，主要发生在前两层（PE 和 CE）。第三层（AH）还在形成中，尚无清晰的代表人物和方法论。

---

## 流派划分

### 流派一：内观派（代表：李继刚）

**工作层：室内设计（Prompt Engineering）**

**核心信念：Prompt 是内心清晰度的投影。**

- 写不好 Prompt，不是因为技巧不够，是因为脑子里那个东西还没立住
- "read in. prompt out." — 输出质量是输入质量的函数
- 压缩表达：一段话→一句话→一个词→一个字。越凝练，在 embedding 空间射得越准（箭 vs 云）
- 角色是压缩包："哲学家"三个字 = 知识域 + 语言风格 + 决策倾向
- "得其意，忘其言" — 语言是障碍，意才是目标
- Lisp 语法不是炫技，是思维的自然形态：函数嵌套 = 思维链

**关键案例：**
- 「汉语新解」— 核心只有5个词，泛化能力极强，因为 attention 无漂移空间。30行 Lisp，一首诗
- 「升维公式」— 用辩证法结构（正题→反题→公因式→新变量→综合）驱动模型做认知跃迁

**这个流派的特质：**
- 最难被"学会"——你学不了他的 Prompt，只能去活你自己的 read in
- 不是技法，是修为
- 写 Prompt 的过程就是在压缩自己的大脑
- Prompt 的起点是面向自己的（脑海中那个清晰的东西）

**局限/张力：**
- 高度个人化，不可复制，不可规模化
- 适合个体创作者，不适合团队协作
- 当模型足够强时，是否还需要这种程度的人类凝练？

---

### 流派二：工程派（代表：宝玉 / Anthropic）

**工作层：建筑设计（Context Engineering）**

**核心信念：Prompt 是持续迭代的工程产物。**

**个人实践（宝玉）：**

- Prompt 不是写出来的，是改出来的
- 让 Agent 自己发现问题，人只做质量判官和方向指挥
- 所有产物持久化：分析、提示词、初稿、审校意见全部保存为文件
- 提示词即代码：可检查、可修改、可复用、可版本控制
- 从串行到并行，一致性靠共享的提示词文件保证
- Skill 要原子化，每个只做一件事，用 AGENTS.md 编排成工作流
- "站在 Agent 角度设计" — Prompt 的读者不是人，是 Agent
- baoyu-translate v1.56.1：156次迭代，几百行 Markdown + 脚本 + 引用文件。一套施工图纸

**组织形态（Anthropic）：**

- Skill 不只是 Markdown 文件，是包含脚本、资产、数据、配置钩子的完整文件夹
- 9 大类型分类框架：从 API 参考到运维手册
- Gotchas 章节：信息密度最高的部分，由真实踩坑积累
- 分发与治理：从代码库提交到内部插件市场
- 衡量效果：用 PreToolUse 钩子记录使用情况

**关键案例：**
- 宝玉翻译 Skill 的四次提示词传递重构（从子 Agent 自己找文件 → 主 Agent 整合传参 → 保存为文件 → 拆分共享上下文与任务指令）
- 「去 AI 味」的核心洞察："所有去 AI 味提示词都是错的" → 你不能靠"不要什么"来定义风格，你要让 AI 尝过你的菜
- 写作风格 Skill：不是一天写出来的，是改了几十篇文章迭代出来的
- Anthropic 内部数百个 Skill 的分类与实战经验

**有意思的细节：** 宝玉翻译 Skill 的翻译原则里有 "Meaning over words"、"Emotional fidelity"——这不是工程规范，这是内观。工程派在工程框架里嵌入了内观派的洞察，只不过用规则清单而不是压缩词。

**这个流派的特质：**
- 可复制、可传授、可规模化
- 有方法论：发现问题 → 提供对比样例 → 指明方向 → 让 Agent 自己优化
- "Agent 比你更懂怎么写好提示词，但你要告诉它方向"
- 从个人匠人到团队工业化的自然延伸
- Prompt 的起点是面向模型的（站在 Agent 角度设计）

**局限/张力：**
- 重流程，可能丢失直觉
- 当你有一套完美的迭代流程时，你是否还需要"脑海中那个清晰的东西"？
- 方法论能出产 80 分的 Prompt，但最后 20 分可能需要内观派那种凝练

**两个流派的核心对立：**
- 内观派说 Prompt 的起点是面向自己（脑海中的清晰画面）
- 工程派说 Prompt 的设计要面向模型（站在 Agent 角度）
- 一个向内，一个向外

---

### 我们的位置：光谱之间

我们不是第三极，而是站在两者之间的张力地带。

**偏内观的一面：**
- 《AI大法官》的核心追问——"你以什么认知结构在理解我"——这是李继刚式的问题
- intent-writer 要求先压缩出目标不变量，和"压缩表达"是同一种冲动
- "这确实发生了"这种最小化确认，是内观者的语感

**偏工程的一面：**
- CLI 接口、hard parsing rules、references/ 目录——纯工程实现
- "synthesizing from history, not generating from vibes"——工程师的信条
- 渐进式信息披露、版本化、可复用

**用工程的手段去逼近内观的问题。** 不是第三派，是两边引力都在。

### 开头：一个问题

### 第一幕：内观派
- 从「汉语新解」的 5 个核心词切入：30行 Lisp，一首诗
- 展开李继刚的压缩哲学
- 引用直播：read in, prompt out / 箭与云 / "得其意忘其言"
- 论述：这个流派的力量和边界——室内设计的极致

### 第二幕：工程派
- 从宝玉翻译 Skill v1.56.1 切入：几百行 Markdown，一套施工图纸
- 同样是"写 Prompt"，一个是一首诗，一个是一套施工图纸——两者都有效，有效的原因完全不同
- "所有去 AI 味提示词都是错的" 作为内观时刻
- Anthropic 的组织实践作为规模化形态
- 论述：可复制性 vs 个人性——建筑设计需要团队协作

### 转折：两个流派的对话
- 内观派会说工程派"模其形未得其意"
- 工程派会说内观派"不可传授不可规模化"
- 但工程派的翻译原则里有 "Meaning over words"——它在工程框架里嵌入了内观
- 面向自己 vs 面向模型，向内 vs 向外

### 尾声：光谱与尺度
- 不是裁判，是辨认：你面对的问题是什么尺度的？
- 室内设计 / 建筑设计 / 城市规划——每层都有不可替代性
- 第三层（Agent Harness / 城市规划）还在形成中
- 可能的落点：真正的问题不是"哪个流派更好"，而是"你面对的是什么尺度的问题"，以及"你想和 AI 建立什么样的关系"
