# Prompt 的三种信仰：一篇关于流派的观察

## 结构稿 v0.1

---

## 文章主旨

Prompt 写作已经分化出了不同的流派。这些流派之间的差异不在于格式（Markdown vs Lisp vs YAML）、不在于容器（单条提示词 vs Skill 文件夹），而在于一个更根本的分歧：**你相信 Prompt 本质上是什么？**

这篇文章不是技巧清单，是一份田野观察。

---

## 流派划分

### 流派一：内观派（代表：李继刚）

**核心信念：Prompt 是内心清晰度的投影。**

- 写不好 Prompt，不是因为技巧不够，是因为脑子里那个东西还没立住
- "read in. prompt out." — 输出质量是输入质量的函数
- 压缩表达：一段话→一句话→一个词→一个字。越凝练，在 embedding 空间射得越准
- 角色是压缩包："哲学家"三个字 = 知识域 + 语言风格 + 决策倾向
- "得其意，忘其言" — 语言是障碍，意才是目标
- Lisp 语法不是炫技，是思维的自然形态：函数嵌套 = 思维链

**关键案例：**
- 「汉语新解」— 核心只有5个词，泛化能力极强，因为 attention 无漂移空间
- 「升维公式」— 用辩证法结构驱动模型做认知跃迁

**这个流派的特质：**
- 最难被"学会"——你学不了他的 Prompt，只能去活你自己的 read in
- 不是技法，是修为
- 写 Prompt 的过程就是在压缩自己的大脑

**局限/张力：**
- 高度个人化，不可复制，不可规模化
- 适合个体创作者，不适合团队协作
- 当模型足够强时，是否还需要这种程度的人类凝练？

---

### 流派二：工程派（代表：宝玉 / Anthropic）

**核心信念：Prompt 是持续迭代的工程产物。**

**个人实践（宝玉）：**

- Prompt 不是写出来的，是改出来的
- 让 Agent 自己发现问题，人只做质量判官和方向指挥
- 所有产物持久化：分析、提示词、初稿、审校意见全部保存为文件
- 提示词即代码：可检查、可修改、可复用、可版本控制
- 从串行到并行，一致性靠共享的提示词文件保证

**组织形态（Anthropic）：**

- Skill 不只是 Markdown 文件，是包含脚本、资产、数据、配置钩子的完整文件夹
- 9 大类型分类框架：从 API 参考到运维手册
- Gotchas 章节：信息密度最高的部分，由真实踩坑积累
- 分发与治理：从代码库提交到内部插件市场
- 衡量效果：用 PreToolUse 钩子记录使用情况

**关键案例：**
- 宝玉翻译 Skill 的四次提示词传递重构
- 「去 AI 味」的核心洞察："所有去 AI 味提示词都是错的" → 你不能靠"不要什么"来定义风格，你要让 AI 尝过你的菜
- 写作风格 Skill：不是一天写出来的，是改了几十篇文章迭代出来的
- Anthropic 内部数百个 Skill 的分类与实战经验

**这个流派的特质：**
- 可复制、可传授、可规模化
- 有方法论：发现问题 → 提供对比样例 → 指明方向 → 让 Agent 自己优化
- "Agent 比你更懂怎么写好提示词，但你要告诉它方向"
- 从个人匠人到团队工业化的自然延伸

**局限/张力：**
- 重流程，可能丢失直觉
- 当你有一套完美的迭代流程时，你是否还需要"脑海中那个清晰的东西"？
- 方法论能出产 80 分的 Prompt，但最后 20 分可能需要内观派那种凝练

---

### 流派三：合约派（代表：我们）

**核心信念：Prompt 是认知关系的前置合约。**

- 在对话发生之前，先把双方的认知边界说清楚
- Prompt 像 API：有明确入口、可预测行为、最小接口暴露面、按需展开的实现细节
- 渐进式信息披露：SKILL.md 保持精简，细节外包到 references/，按需加载
- "synthesizing from history, not generating from vibes" — 从成功的历史中蒸馏，不从感觉中生成
- Hard parsing rules — 不是约束模型的自由度，是在定义双方的契约

**关键案例：**
- intent-writer：先把 Goal + Target state invariants 压缩成 INTENT.md
- skill-distiller：从对话历史中提取 initial attempt → every user correction → final accepted artifact
- 《AI大法官》中的核心追问："你是在哪一刻，开始替我把感受提前安放进语言？"

**这个流派的特质：**
- 技术和哲学的交叉点：用工程方法处理认知问题
- 关注的不是模型输出什么，而是模型以什么认知结构在理解你
- 最终裁决是最小化的确认："这确实发生了"

**局限/张力：**
- 过度设计的风险：当合约太完备时，留给意外和涌现的空间变小
- 并非所有场景都需要前置合约——有时候直接说话就够了
- 这种写法对作者的系统设计能力要求极高

## 文章结构

### 开头：一个问题
（叙事开头，你来写。核心问题：为什么同样一个"写 Prompt"的行为，不同的人做出了本质不同的事？）

### 第一幕：内观派
- 从「汉语新解」的 5 个核心词切入
- 展开李继刚的压缩哲学
- 引用他的直播：read in, prompt out / 箭与云的比喻 / "得其意忘其言"
- 论述：这个流派的力量和边界

### 第二幕：工程派
- 从宝玉翻译 Skill 的迭代史切入
- "所有去 AI 味提示词都是错的" 作为转折点
- Anthropic 的组织实践作为这个流派的规模化形态
- 论述：可复制性 vs 个人性的张力
- 两个流派的对话：内观派会说工程派"模其形未得其意"，工程派会说内观派"不可传授不可规模化"

### 第三幕：合约派
- 从《AI大法官》中的一个追问切入："你是在哪一刻开始替我安放感受的？"
- 展开 Prompt 作为认知合约的理念
- 接口设计 + 渐进披露 + 从历史中蒸馏
- 论述：这个流派站在前两者的交叉点上

### 结尾：不是结论
（不做裁判，不说谁对谁错。可能的方向：这三种信仰分别对应了人与 AI 关系的三种想象——你是画师、你是工程师、你是审问者。而真正的问题也许是：你想和 AI 建立什么样的关系？）
